🧠 Key Concepts

Class = Blueprint, Object = Real Entity

A class acts as a blueprint (like an architectural drawing before a house is built) — it defines the structure. An object is the actual real entity created from that blueprint (the real house). Example: an SBIUser class is the blueprint; each actual customer who logs in becomes a real SBIUser object holding their specific details.

Why We Create Objects

To represent a real-world entity (a person, a product, a payment, an employee) inside a program, you create an object of the relevant class. One class can be used to create many objects — e.g. millions of SBI customers all use the same SBIUser class, but each gets their own unique object holding their own data.

Class Members Recap: Fields, Methods, Constructors
  • Fields (data/attributes/state) — e.g. name, userId, password, balance
  • Methods (behavior) — what the object can do, e.g. deposit, withdraw
  • Constructor (new) — special member responsible for creating the object
What is a Constructor?

"Construct" means to create/build. A constructor is a special block of code, with the same name as the class, that runs when an object is created — it's responsible for constructing (creating) the object, and can also be used to initialize its field values.

Constructor vs Method — Key Difference

Unlike a method, a constructor has no return type (not even void) and must be named exactly the same as the class.

Creating an Object — the new Keyword

The new keyword instructs Java to create a new object by calling a constructor. Without new, the compiler/JVM would have no clear instruction to build an object — new makes the intent explicit.

SBIUser user = new SBIUser();

Here, SBIUser (left) is the type, user is the reference variable name, and new SBIUser() creates the actual object and calls its constructor.

No-Argument Constructor

A constructor that takes no parameters. When called, it creates the object but — unless you write initialization logic inside it — fields are left at their default values (0 for numeric primitives, null for object references like String).

Parameterized Constructor

A constructor that accepts arguments, used to initialize field values at the time of object creation. A class can have multiple constructors (constructor overloading) — Java picks the correct one to call based on the number and type of arguments supplied.

Local Parameters vs Instance Fields — Naming Convention

To avoid confusion between a constructor's local parameter names and the class's instance fields, an underscore prefix convention was used for parameters (e.g. _name) so it's clear which is temporary/local and which is the object's actual state. Alternatively, this.fieldName can be used to explicitly refer to the current object's instance variable.

Instance Variables — Specific to Each Object

Even though instance variables are declared once in the class, each object gets its own independent copy of their values. Creating two objects of the same class (e.g. one for "Abhinav", one for "Asis") results in two separate sets of field values in memory — changing one object's data does not affect the other's.

Reference Variable (Stack) vs Actual Object (Heap)

The reference variable (e.g. user) is a local variable stored in the Stack. The actual object it points to is stored in the Heap. The reference variable simply holds the address of the real object in the heap — similar to how every real person has a unique physical address.

Default Constructor (Compiler-Provided)

If a class has no constructor written at all, the Java compiler automatically adds a hidden no-argument default constructor, which is why new ClassName() still works even without an explicit constructor. However, if you write even one constructor yourself (e.g. with parameters), the compiler no longer provides the default one — so calling a no-argument constructor after that will fail unless you explicitly define one too.

💻 Code Snippets / Examples

Class with fields, a no-argument constructor, and a parameterized constructor:

package com.sbi;

class SBIUser {

    private String name;
    private String userId;
    private String password;
    private int balance;

    // No-argument constructor
    public SBIUser() {
        System.out.println("Constructor called, new object created");
    }

    // Parameterized constructor
    public SBIUser(String _name, String _userId, String _password, int _balance) {
        name = _name;
        userId = _userId;
        password = _password;
        balance = _balance;
    }

}

Creating objects — no-argument vs parameterized:

class Main {

    public static void main(String[] args) {
        SBIUser user1 = new SBIUser(); // calls no-argument constructor
        // fields will be at default values: null, null, null, 0

        SBIUser user2 = new SBIUser("Abhinav", "Abhi123", "test123", 10000);
        // fields initialized: name="Abhinav", userId="Abhi123", password="test123", balance=10000

        SBIUser user3 = new SBIUser("Asis", "Asis123", "pass456", 20000);
        // separate object — its own independent copy of field values
    }

}

Using "this" for clarity (alternative to underscore convention):

public SBIUser(String name, String userId, String password, int balance) {
    this.name = name;       // this.name refers to the current object's field
    this.userId = userId;
    this.password = password;
    this.balance = balance;
}

What happens without a written constructor (compiler adds default one):

class Payment {
    // no constructor written here
}

// Still works — compiler silently adds a no-argument default constructor:
Payment payment = new Payment();

Why this fails once you add a parameterized constructor:

class Payment {
    public Payment(int amount) {
        // ...
    }
}

// This will NOT compile — compiler no longer provides a default no-arg constructor:
Payment payment = new Payment(); // ERROR

📌 Important Points / Common Mistakes

  • Constructor name must exactly match the class name, and it has no return type.
  • Instance variable values are unique per object — don't confuse shared field declarations with shared values.
  • If you write any parameterized constructor, the compiler no longer auto-generates a default no-argument constructor — write one explicitly if both are needed.
  • Reference variables live on the Stack; actual objects live on the Heap.
  • Real understanding of instance variables and object creation matters more than chasing newer Java version features — solid fundamentals matter most.
  • Use debugging (breakpoints, stepping through code) to actually observe how objects and their field values are created and initialized — this builds much deeper understanding than reading code alone.
