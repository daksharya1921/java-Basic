Class 28 — Upcasting, Downcasting & Method Overriding

🎯 Topics Covered

  • Upcasting — parent class reference pointing to a child object (automatic)
  • Downcasting — child class reference pointing to a parent-typed object (explicit, required)
  • Why upcasting/downcasting matters — generic "factory" style methods returning Object
  • ClassCastException — when downcasting fails at runtime
  • Method Overriding — changing a parent class method's behavior in the child class
  • Rules for overriding: exact same method signature required
  • @Override annotation — purpose and optional nature
  • Java method resolution order: child class → superclass → Object class

🧠 Key Concepts

Upcasting — Parent Reference to Child Object (Automatic) 
Employee e = new Manager(); // Manager IS-A Employee

A parent class reference can point to a child class object. This is called upcasting, and it happens automatically in Java — no explicit cast syntax is needed. This only works when a genuine parent-child (inheritance) relationship exists between the two classes.

Downcasting — Child Reference to Parent-Typed Object (Explicit) 
Employee e = new Manager();
Manager m = (Manager) e; // explicit downcast required

The reverse — assigning a reference that's currently typed as the parent class to a more specific child class variable — is called downcasting, and it does not happen automatically. You must explicitly write the target type in parentheses to tell the compiler you intend to do this.

Why Downcasting is Needed — Generic "Factory" Methods

A common real-world pattern: a method designed to return many different types of objects (e.g. based on a name/key supplied by the caller) can't declare a specific return type for each possible class — instead, it declares the return type as Object (the common ancestor of everything). The caller then needs to downcast the returned Object back to the specific type they expect to actually use its specific methods/fields.

ClassCastException

Downcasting only resolves the compile-time type check — it does not guarantee the cast is valid at runtime. If you downcast an object to a type that doesn't match its actual underlying type (no real inheritance relationship to that actual object), Java throws a ClassCastException at runtime. The cast must match the object's true identity, not just look valid syntactically.

Method Overriding — Changing Parent Behavior in the Child

If a child class inherits a method from its parent but wants different behavior, it can redefine that method inside itself with the exact same signature (name, parameters). This is called method overriding — the child's version replaces the parent's version when called on a child object.

Rules for a Valid Override

The method in the child class must have the exact same name as the parent's method — Java is case-sensitive, so even a single differing letter's case makes it a completely different (unrelated) method, not an override.

@Override Annotation — Optional Safety Check

@Override (introduced in Java 5) is an optional annotation you can place above a method to tell the compiler "I intend this to override a parent method." If there's a mistake (e.g. a typo in the method name, meaning it doesn't actually match any parent method), the compiler will flag an error — catching accidental non-overrides that would otherwise silently just become a new, unrelated method.

Method Resolution Order

When a method is called on an object, Java first checks if it's defined in that object's own class. If not found there, it checks the superclass, and if still not found, continues up to the Object class at the top. This is how overriding takes priority — the child's own version is found first.

"Functionality" = What's Inside the Curly Braces

When discussing overriding "changing the functionality" of a method, this simply refers to changing the code/logic written inside the method's curly braces — the actual implementation, not the method's name or signature.

💻 Code Snippets / Examples

Upcasting and downcasting basics:

class Employee { }
class Manager extends Employee { }

class Driver {
    public static void main(String[] args) {
        Employee e = new Manager(); // upcasting — automatic

        Manager m = (Manager) e;    // downcasting — explicit cast required
    }
}

Method overriding — child changes parent's behavior:

class Product {
    public void displayProductDetail() {
        System.out.println("Generic product details");
    }
}
class ElectronicProduct extends Product {

    @Override
    public void displayProductDetail() { // exact same signature — this is overriding
        System.out.println("Electronic product details");
    }

}
class Driver {
    public static void main(String[] args) {
        ElectronicProduct ep = new ElectronicProduct();
        ep.displayProductDetail(); // "Electronic product details" — overridden version runs
    }
}

Object factory pattern — returning Object, requiring downcast:

class Product {
    public void displayProductDetail() {
        System.out.println("Product details");
    }
}

class ElectronicProduct extends Product {
    @Override
    public void displayProductDetail() {
        System.out.println("Electronic product details");
    }
}
class ObjectFactory {

    public static Object getObject(String name) {
        Object obj = null;

        if (name.equalsIgnoreCase("product")) {
            obj = new Product();
        } else if (name.equalsIgnoreCase("electronic")) {
            obj = new ElectronicProduct();
        }

        return obj; // returns Object — the common supertype of everything
    }

}
class Driver {
    public static void main(String[] args) {
        Object obj = ObjectFactory.getObject("product");

        // obj.displayProductDetail(); // ERROR — Object class has no such method

        Product p = (Product) obj; // downcast required to use Product-specific methods
        p.displayProductDetail();  // works fine now
    }
}

ClassCastException — downcasting to an unrelated/incorrect type:

class Vehicle { }
class Car extends Vehicle { }
class Bike extends Vehicle { }

Vehicle v = new Car();
Bike b = (Bike) v; // compiles fine, but throws ClassCastException at RUNTIME
                    // because the actual object is a Car, not a Bike — no valid relationship

❓ Doubts / Questions Raised in Class

QuestionAnswer
Does downcasting guarantee success?No — downcasting only satisfies the compiler; if the actual runtime object doesn't match the target type's inheritance chain, it throws a ClassCastException when executed.
Why is @Override useful if it's optional?It acts as a safety net — the compiler checks that a matching parent method actually exists, catching accidental typos or signature mismatches that would otherwise silently create an unrelated new method instead of a true override.
If a child needs both the parent's existing functionality AND new specific functionality, should it override the same method for both?No — write a separate new method for the child-specific functionality; don't try to cram both parent and child logic into a single overridden method meant for one purpose.
Why does a factory-style method return Object instead of a specific type?Because it needs to support returning many different object types based on runtime input — Object is the common ancestor that can represent all of them, with downcasting used by the caller afterward.

📌 Important Points / Common Mistakes

  • Upcasting (parent reference → child object) is automatic; downcasting (child reference → parent-typed object) requires an explicit cast.
  • Downcasting to an incompatible type compiles but throws ClassCastException at runtime — always ensure a valid inheritance relationship exists.
  • Overriding requires an exact match in method name (Java is case-sensitive) and signature — even one differing letter case creates an unrelated method instead.
  • Use @Override to catch accidental non-matching "overrides" at compile time.
  • Java looks for a method first in the object's own class, then the superclass, then Object — this is how overridden methods take precedence.
  • Don't blend unrelated logic into one overridden method — separate distinct responsibilities into separate methods.
