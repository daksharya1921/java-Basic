What Encapsulation Means

A class should not allow other classes to modify its data (fields) directly. Access should instead go through a controlled interface — similar to how a home has boundaries/privacy, and neighbors can't just walk in and interfere, but can interact through agreed, controlled means when needed.

Why Controlling Visibility Matters

If a class exposes its fields as public, any other class can set them to any value — including invalid or nonsensical ones (e.g. a negative account balance). This is a design flaw: the class has no control over what values its own data takes.

The Standard Pattern: Private Fields + Public Getters/Setters
  • Declare fields as private — this prevents other classes from directly reading or modifying them.
  • Provide public getter methods to allow controlled reading of the data.
  • Provide public setter methods to allow controlled modification of the data — with validation logic inside.
Controlled Modification via Setter Logic

Inside a setter (or any method that modifies state, like deposit() or withdraw()), you can add validation — e.g. rejecting negative deposit amounts, or preventing a withdrawal that exceeds the current balance. This is exactly what "modification in a controlled way" means — the class itself decides what changes are valid, rather than blindly accepting whatever the caller supplies.

"Not Well Encapsulated" — A Common Interview/Review Question

A very common interview/code-review scenario: you're shown working code (with public fields being modified directly from another class) and asked "what's the problem with this code?" The correct answer isn't about syntax — it's identifying that the class is not well encapsulated, since other classes can directly modify its data without any validation or control.

Getters/Setters Don't Have to Be Named get/set

While getX()/setX() is the common convention, the underlying idea is just "a controlled method to read/modify data." Domain-specific method names (like deposit() and withdraw() in a banking context) achieve the same encapsulation principle — they're effectively setters with built-in business validation.

Why OOPs Matters for LLD Interviews

Low-Level Design (LLD) interview rounds are largely evaluating how well a candidate applies OOPs concepts — not just whether the code "works" or produces correct output. Many candidates get rejected in LLD rounds despite working code, because they focus only on logic/output and not on whether their design follows encapsulation, abstraction, inheritance, and polymorphism properly.

💻 Code Snippets / Examples

NOT well encapsulated — public field, direct access:

class Account {
    public int balance; // public — no control at all
}
class User {
    public static void main(String[] args) {
        Account account = new Account();
        account.balance = -2000000; // any class can set anything, even invalid values
        System.out.println(account.balance);
    }
}

Well encapsulated — private field with controlled getter/setter:

class Account {
    private int balance;

    public int getBalance() {
        return balance;
    }

    public void setBalance(int balance) {
        if (balance >= 0) { // controlled validation
            this.balance = balance;
        } else {
            System.out.println("Invalid amount");
        }
    }
}
class User {
    public static void main(String[] args) {
        Account account = new Account();
        account.setBalance(-2000); // rejected — invalid amount, balance stays unchanged
        System.out.println(account.getBalance()); // 0 (default, unchanged)
    }
}

Practical example — bank Account with deposit/withdraw (domain-specific setters):

class Account {

    private int balance;
    private String name;

    public Account(int balance, String name) {
        this.balance = balance;
        this.name = name;
    }

    public void deposit(int amount) {
        if (amount > 0) {
            balance = balance + amount;
        }
    }

    public void withdraw(int amount) {
        if (amount <= balance) {
            balance = balance - amount;
        } else {
            System.out.println("Insufficient balance");
        }
    }

    public int checkBalance() {
        return balance;
    }

}
class ATM {
    public static void main(String[] args) {
        Account account = new Account(10000, "Rahul");

        System.out.println(account.checkBalance()); // 10000

        account.withdraw(22000); // rejected: Insufficient balance
        account.withdraw(2000);  // allowed
        System.out.println(account.checkBalance()); // 8000

        account.deposit(2000);
        System.out.println(account.checkBalance()); // 10000

        // account.balance = 999999; // NOT possible — private, compile error
    }
}

📌 Important Points / Common Mistakes

  • Public fields with no validation is a classic encapsulation violation — always flag this in code review scenarios.
  • Declare fields private by default; expose controlled access only through public methods.
  • Add validation logic inside setters/modifier methods rather than allowing blind updates.
  • "Not well encapsulated" is a specific, correct answer to give when asked what's wrong with code that allows direct, uncontrolled field modification.
  • Encapsulation doesn't mean isolating a class from all others — it means controlling how interaction happens.
