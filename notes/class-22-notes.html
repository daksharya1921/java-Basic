Class 22 — String Concatenation Optimization & intern()

🎯 Topics Covered

  • Concatenating two String literals with + — compile-time optimization
  • Concatenating String objects/variables with + — runtime behavior via StringBuilder
  • Object counting practice across various String creation scenarios
  • Why Java provides both content-based (SCP) and identity-based (heap) object creation — practical use cases
  • The intern() method — purpose and behavior
  • String concatenation with non-string types (numbers, null) and left-to-right evaluation

🧠 Key Concepts

Compile-Time Optimization: Literal + Literal

When you concatenate two String literals using +, the Java compiler optimizes this at compile time — it merges them into a single combined literal before the program even runs. As a result, only one object is created in the SCP for the final combined string — exactly as if you'd written the combined string directly.

Runtime Behavior: Variable/Object + Anything

When concatenating using + where at least one operand is a variable/reference (not a plain literal known at compile time), Java cannot optimize this ahead of time — it happens at runtime. Internally, Java uses a StringBuilder to perform the concatenation, and the resulting object is created in the Heap (not the SCP).

Why Both SCP (Content) and Heap (Identity) Creation Exist

Recap/expansion: most real applications favor content-based reuse (SCP) since the same values (city names, product names, etc.) repeat across millions of users — creating a new object every time would waste huge amounts of memory. However, some use cases specifically need object identity — e.g. location-tracking or audit-trail systems where you need to know how many distinct actions/objects were created for a given value, regardless of whether the content repeats. In these cases, new keyword object creation (always a fresh heap object) is the right tool.

The intern() Method

intern() is used to move/link a heap-created String object to the SCP. When called on a String:

  • If a String with the same content already exists in the SCP, the reference now points to that existing SCP object (and the original heap object becomes unreferenced/orphaned, eligible for garbage collection).
  • If no matching object exists in the SCP yet, the content is copied into the SCP as a new object, and the reference now points to that new SCP object.

Purpose: memory optimization — letting you take a heap-created (e.g. via new or concatenation) String and "promote" it into the reusable SCP pool if desired.

Orphaned Objects and Garbage Collection

When a reference is redirected (e.g. via intern()) away from a heap object, and nothing else refers to that old heap object, it becomes orphaned — eligible to be reclaimed by the garbage collector.

Concatenating Strings with Numbers and null

The + operator evaluates left to right. Once a String is involved anywhere in the chain, everything after it in that evaluation gets treated as a String and appended — including numbers (converted to their string form) and even null (printed literally as the text "null", not a NullPointerException) — as long as a String starts the concatenation early enough in the expression.

💻 Code Snippets / Examples

Compile-time optimization — literal + literal:

String s = "Codewala" + " Academy";
// Compiler merges this into "Codewala Academy" BEFORE runtime
// Only ONE object created in SCP — same as writing:
// String s = "Codewala Academy";

Runtime concatenation — object/variable involved, uses StringBuilder, goes to heap:

String s1 = new String("Hello");
String s2 = s1 + " World";
// s1 is a reference (not a compile-time-known literal)
// Concatenation happens at RUNTIME using StringBuilder internally
// Result object "Hello World" is created in the HEAP, not SCP

String s5 = "Hello World"; // literal — SCP object
System.out.println(s2 == s5); // false — s2 is in heap, s5 is in SCP, different objects
System.out.println(s2.equals(s5)); // true — same content

Using intern() to link a heap object to the SCP:

String s1 = new String("Hello"); // heap object (+ "Hello" also exists/created in SCP)
String s2 = "Hello";             // SCP object

System.out.println(s1 == s2); // false — s1 is heap, s2 is SCP

String s3 = s1.intern(); // "Hello" already exists in SCP -> s3 now points to that SCP object
System.out.println(s3 == s2); // true — both now point to the same SCP object

intern() when the content doesn't exist yet in SCP — gets copied in:

String s10 = "Codewala";
String combined = s10.concat(" Academy"); // heap object "Codewala Academy" (via runtime concat)

String internedResult = combined.intern();
// "Codewala Academy" doesn't exist in SCP yet -> it gets COPIED into SCP
// internedResult now refers to this new SCP object

String concatenation with numbers and null — left to right evaluation:

System.out.println("Value: " + 50 + "Sachin"); // "Value: 50Sachin"
// starts as String -> everything appended as string from left to right

System.out.println(50 + 60 + "Sachin"); // "110Sachin"
// 50 + 60 evaluated as numbers FIRST (both are int, no string yet) = 110
// then "Sachin" appended as string -> "110Sachin"

String name = null;
System.out.println("Sachin" + name); // "Sachinnull"
// null is printed literally as text "null" when concatenated with a String

❓ Doubts / Questions Raised in Class

QuestionAnswer
If objects in the SCP have no reference anymore, why don't they get garbage collected like heap objects?SCP objects persist for reusability — since the same content is likely to be needed again by other parts of a large-scale application, keeping them avoids recreating identical objects repeatedly; SCP objects are generally retained until class unloading, not eagerly garbage collected like typical heap objects.
Why not always use identity (new keyword) objects instead of SCP, for safety/tracking purposes?Because most real-world data (city names, product names, etc.) genuinely repeats across huge numbers of users — using new for everything would cause massive unnecessary memory usage; identity-based creation is reserved for specific use cases like tracking distinct actions/movements.
What is the actual purpose of intern() if literals already go to SCP automatically?intern() is specifically for Strings that were NOT created as literals (e.g. via new or concatenation, which land in the heap) — it lets you deliberately move/link them into the SCP for reuse, rather than leaving them as separate heap objects.
Would storing sensitive data (like credit card numbers) in the SCP be a security concern?Not really a distinguishing factor — sensitive/unique data like credit card numbers wouldn't benefit from SCP's reuse mechanism anyway (since they're unique per customer, not repeated), so they'd typically be handled via heap objects (and masked/secured through other means) rather than SCP.

📌 Important Points / Common Mistakes

  • Literal + literal concatenation is optimized at compile time into a single SCP object.
  • Concatenation involving any variable/object reference happens at runtime via StringBuilder, and the result lands in the heap.
  • intern() is the tool to move a heap-created String into the SCP (reusing an existing match, or copying it in as new).
  • String concatenation evaluates left to right — numeric addition only happens before a String appears in the expression; after that, everything becomes string concatenation.
  • Concatenating null with a String prints the literal text "null" — it does not throw a NullPointerException.
  • Don't assume identity-based (new keyword) object creation is "safer" by default — it's a deliberate design choice for specific tracking/identity use cases, not a general best practice.
