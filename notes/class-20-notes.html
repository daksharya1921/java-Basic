🧠 Key Concepts

Private Constructors

A constructor can be marked private. This means it can only be called within the same class — no other class can create an object of that class using new from outside. This is different from a private method/field only in that it restricts object creation itself.

Singleton Design Pattern (Introduced)

A common use case for a private constructor: ensuring a class can have only one instance (object) throughout the entire application. The class itself controls object creation internally — if an object already exists, it returns the existing one instead of creating a new one. This is called the Singleton design pattern.

Refactoring Constructor Logic into IIB (Best Practice)

When multiple overloaded constructors all need to run the same line of setup logic (e.g. incrementing an object counter), duplicating that line in every constructor is inefficient and error-prone at scale. The better approach: move that shared logic into an Instance Initialization Block (IIB), since IIB runs automatically for every constructor/object creation — eliminating duplication.

String is a Class, Not a Primitive

Unlike int, boolean, char, etc. (primitive data types), String is a class in Java — specifically part of the java.lang package. This means a String is always a non-primitive (object) type, and creating a String means creating an object.

String Constant Pool (SCP)

A special reserved area used specifically for String objects created via literals. Before Java 8, SCP was part of PermGen; from Java 8 onward, it is part of the Heap.

Two Ways to Create a String
  • String literal: String city = "Bangalore"; — no new keyword, always goes to the String Constant Pool
  • Using new keyword: String city = new String("Bangalore"); — creates a separate object (typically directly in the heap, outside SCP) — covered further in upcoming classes
SCP Deduplication Rule

When creating a String via a literal, Java first checks whether a String with the exact same content already exists in the SCP. If it does, no new object is created — the new reference simply points to the existing object. If it doesn't exist, a new object is created in the SCP. This means there is never more than one object with the same content in the SCP at a time.

Case Sensitivity Matters

Since Java is case-sensitive, "Bangalore" and "bangalore" (different capitalization) are treated as different content — each would get its own separate object in the SCP.

Why Strings Get Special Treatment

Strings are used extremely heavily across virtually all applications (roughly 80–90% of fields in real-world systems are String type — names, addresses, descriptions, etc.). To avoid excessive memory usage from creating duplicate objects with identical content repeatedly, Java optimizes heap usage specifically for Strings via the SCP deduplication mechanism.

String Immutability — What It Actually Means

Immutable does not mean "you cannot attempt to change it." It means: once created, an existing String object's content itself can never be altered. If you try to modify a String's value (e.g. via concatenation), Java creates a brand new object with the new value — the original object remains unchanged and untouched.

Reassignment Needed to "See" the Modified Value

Since modifying a String actually creates a new object rather than changing the original, if you don't reassign the reference variable to point to that new object, the reference still points to the old (unchanged) object — so printing it will still show the original value.

Comparing References with ==

== checks whether two references point to the same object address in memory — not whether their content is equal. Two String literals with identical content will return true with == (because SCP deduplication means they point to the same object); two Strings with different content (or created differently) may return false even if a human would consider them related.

💻 Code Snippets / Examples

Private constructor — restricts object creation to within the class:

class Payment {

    private String paymentRefId;
    private int amount;
    private String status;
    private String response;

    private Payment(String _refId, int _amount, String _status, String _response) {
        paymentRefId = _refId;
        amount = _amount;
        status = _status;
        response = _response;
    }

}
class Account {

    public static void main(String[] args) {
        Payment p = new Payment("REF1", 13400, "SUCCESS", "OK");
        // ERROR: constructor Payment() has private access in Payment
        // cannot create object from outside the class
    }

}

Refactoring: duplicate constructor logic moved to IIB:

// BEFORE (duplicated in every constructor — not ideal)
class Viewer {
    public static int liveUsers = 0;

    public Viewer() {
        liveUsers = liveUsers + 1; // duplicated logic
    }

    public Viewer(String paymentRef) {
        liveUsers = liveUsers + 1; // duplicated logic
    }
}

// AFTER (moved to IIB — runs automatically for every constructor)
class Viewer {
    public static int liveUsers = 0;

    {
        liveUsers = liveUsers + 1; // shared setup, no duplication
    }

    public Viewer() {
    }

    public Viewer(String paymentRef) {
    }
}

Creating Strings — literal vs new, and SCP deduplication:

String city = "Bangalore"; // literal — created in SCP (1st object)
String india = "India";    // literal — created in SCP (2nd object)
String c = "Bangalore";    // same content — reuses existing object, no new object created

System.out.println(city == c); // true — both point to the same SCP object

Case sensitivity creates a separate object:

String city = "Bangalore"; // capital B
String c = "bangalore";    // lowercase b — different content

System.out.println(city == c); // false — different objects, different addresses

Immutability — modifying a String creates a new object:

String address = "BTM"; // object 1: "BTM" in SCP

address = address + " Bangalore";
// "BTM" itself is NOT modified — a NEW object "BTM Bangalore" is created
// "address" is REASSIGNED to point to this new object

System.out.println(address); // BTM Bangalore

Common misunderstanding — value not changing without reassignment:

String address = "BTM";
address.concat(" Bangalore"); // creates a new object, but result is NOT captured/assigned

System.out.println(address); // still prints "BTM" — original object untouched,
                              // new object created but nothing refers to it

❓ Doubts / Questions Raised in Class

QuestionAnswer
In which scenario would you use a private constructor?When you want to restrict object creation to only within that class itself — e.g. the Singleton design pattern, where only one instance should ever exist.
Which approach for counting created objects is "correct" if both give the right output?Moving the shared logic into IIB is the better approach — it avoids code duplication across multiple constructors, which matters greatly at scale (e.g. millions of object creations).
Does immutable mean you can't even try to change a String?No — you can attempt to modify it, but the original object stays unchanged; the attempt instead produces a brand new object, which you must capture/reference to actually use.
Why does printing a String after calling a method like concat() sometimes show no change?Because the method returns a new object with the modified value, but if that return value isn't reassigned to a variable, nothing refers to the new object — the original reference still points to the unchanged original.

📌 Important Points / Common Mistakes

  • A private constructor restricts object creation to within the declaring class only.
  • Prefer IIB over duplicating identical setup code across multiple constructors.
  • String is a class (non-primitive), not a primitive data type.
  • String literals are deduplicated in the SCP — identical content never creates a duplicate object.
  • Java's case sensitivity means differently-cased strings are different content and get separate objects.
  • String objects are immutable — any "modification" actually creates a new object; you must reassign to see the new value.
  • == compares object references/addresses, not content — be careful not to confuse this with content equality (covered further with .equals() elsewhere).
