🧠 Key Concepts

Recap: Overriding Rules Summary
  • Inheritance is compulsory (overriding happens between parent and child, never within the same class)
  • Method name must be exactly the same
  • Parameters: same order, type, and number
  • Return type: same, or covariant
  • Access modifier: same or wider (never narrower)
Private Methods Cannot Be Overridden (Recap)

Overriding requires the method to be visible in the child class — private methods aren't, since private scope is limited strictly to the declaring class.

Static Methods Cannot Be Truly Overridden — Method Hiding

Static methods belong to the class, and which static method gets called is resolved at compile time — before the program even runs. True overriding, by contrast, is resolved at runtime based on the actual object. Because of this timing mismatch, if a child class declares a static method with the same signature as a parent's static method, this is not overriding — it's called method hiding. The parent's static method becomes hidden (inaccessible via the child reference) rather than replaced/overridden.

Method Overloading — Definition

Overloading means having multiple versions of the same method name within the same class — like different "flavors" of the same method, each uniquely identifiable by its parameter signature.

The One Rule for Valid Overloading

Within the same class, the method name must be the same, but at least one of the following must differ:

  • Number of arguments, OR
  • Type of arguments, OR
  • Order of arguments

Return type, access modifier, and static keyword do NOT matter for overloading — they can be anything and it's still a valid overload, as long as the parameter signature differs in at least one of the three ways above.

Overloading vs Overriding — Key Difference
OverloadingOverriding
Happens WITHIN the same classHappens BETWEEN parent and child classes
No inheritance requiredInheritance is compulsory
Parameters must differParameters must be identical
Why Overloading Exists — Flexibility for the Caller

Overloading lets callers use whichever version of a method fits the data they actually have available. Real-world example: a registration system might allow registering via mobile+OTP, OR via Aadhaar, OR via DL/Voter ID, OR via a bank ATM PIN — since not everyone has every type of ID. Each option is an overloaded version of a "doRegistration" method, and the appropriate one gets called based on what arguments are supplied.

Polymorphism — The Real-World Analogy

Polymorphism means the same entity behaves differently depending on the situation/context. Example: the same person is a "student" in class, a "tenant" at their PG, and a "passenger" at the airport — same person, different behavior based on context. In Java, this translates to: the same method call behaves differently depending on the actual object type.

Runtime Polymorphism — Requires Overriding + Inheritance

Runtime polymorphism (also called dynamic method dispatch or late binding) is achieved through method overriding combined with upcasting (a parent reference pointing to different child objects). Which overridden version of a method actually runs is decided at runtime, based on the actual object type — not the reference variable's declared type.

Two-Phase Method Resolution: Compile Time vs Runtime
  • Compile time: The compiler only checks whether the method exists somewhere accessible on the reference type (the declared type of the variable) — it doesn't know or care what the actual object will be at runtime.
  • Runtime: JVM looks at the actual object that was created and calls that object's version of the method (its own override, if one exists — otherwise falls back up the inheritance chain to the parent's version).
Method Lookup Falls Back Up the Inheritance Chain

If the actual object's class doesn't have its own override of the called method, Java looks in its parent class, then that parent's parent, and so on — all the way up to Object class if needed — calling the first matching version it finds.

💻 Code Snippets / Examples

Static method hiding — NOT true overriding:

class Delivery {
    static void method1() {
        System.out.println("Delivery static method");
    }
}
class ExpressDelivery extends Delivery {
    static void method1() { // this HIDES the parent's static method, doesn't override it
        System.out.println("ExpressDelivery static method");
    }
}
Delivery.method1();        // "Delivery static method"
ExpressDelivery.method1(); // "ExpressDelivery static method"
// Each is resolved at compile time based on the class name used — no runtime decision involved

Method overloading — multiple versions of doRegistration():

class UserRegistration {

    public void doRegistration(String mobile, int otp) {
        System.out.println("Registered via mobile + OTP");
    }

    public void doRegistration(String aadhaar) {
        System.out.println("Registered via Aadhaar");
    }

    public void doRegistration(int atmPin, String cardNumber) {
        System.out.println("Registered via ATM PIN + card");
    }

    public void doRegistration(String dln, String voterId) {
        System.out.println("Registered via DL + Voter ID");
    }

}
class Driver1 {
    public static void main(String[] args) {
        UserRegistration ur = new UserRegistration();
        ur.doRegistration("9876543210", 1234);         // calls (String, int) version
        ur.doRegistration(1234, "ATM123456");            // calls (int, String) version — different order
    }
}

Runtime polymorphism — full example:

class Delivery {
    public String estimateTime(String source, String destination) {
        return "Delivery: standard estimate";
    }
}
class OneDayDelivery extends Delivery {
    @Override
    public String estimateTime(String source, String destination) {
        return "OneDayDelivery: 1 day estimate";
    }
}
class ExpressDelivery extends Delivery {
    @Override
    public String estimateTime(String source, String destination) {
        return "ExpressDelivery: 2 hour estimate";
    }
}
class Driver {
    public static void main(String[] args) {
        Delivery del = new OneDayDelivery(); // reference type: Delivery, actual object: OneDayDelivery
        System.out.println(del.estimateTime("Bangalore", "Delhi"));
        // Compile time: checks estimateTime exists on Delivery type — OK
        // Runtime: actual object is OneDayDelivery -> calls OneDayDelivery's version
        // Output: "OneDayDelivery: 1 day estimate"

        del = new ExpressDelivery(); // same reference, different actual object now
        System.out.println(del.estimateTime("Bangalore", "Delhi"));
        // Output: "ExpressDelivery: 2 hour estimate" — decision changes based on actual object
    }
}

Method lookup falls back to parent if not overridden:

Delivery del = new OneDayDelivery();
del.hashCode(); // OneDayDelivery doesn't define hashCode(), Delivery doesn't either
                 // -> Java goes up to Object class and calls Object's hashCode()

Compilation fails if the method doesn't exist on the reference type, even if the actual object has it:

class Delivery { } // no estimateTime() method here at all

class OneDayDelivery extends Delivery {
    public String estimateTime(String s, String d) {
        return "1 day";
    }
}
Delivery del = new OneDayDelivery();
del.estimateTime("A", "B"); // COMPILE ERROR — estimateTime() not found on Delivery (reference type)
                            // even though the actual object (OneDayDelivery) has it

❓ Doubts / Questions Raised in Class

QuestionAnswer
Can a static method be overridden?No — it's called method hiding, not overriding, since static methods are resolved at compile time, not runtime.
Does return type matter for overloading?No — return type, access modifier, and static/non-static status can all vary freely; only the parameter signature (order/type/number) determines whether it's a valid overload.
Why is runtime polymorphism called "runtime"?Because the decision of which overridden method to actually call is made at runtime, based on the actual object type — not decided during compilation.
What happens if the reference type doesn't have the method being called, even if the actual object does?Compilation fails — the compiler only checks the reference (declared) type, not the actual runtime object.
What happens if neither the actual object's class nor its immediate parent has the called method?Java continues searching up the inheritance chain until it finds a match, ultimately reaching Object class if needed.

📌 Important Points / Common Mistakes

  • Static methods are hidden, not overridden, in child classes — resolved at compile time based on the class name used to call them.
  • Overloading requires differing parameters (order/type/number); return type and modifiers don't matter.
  • Overloading = within a class; Overriding = between parent and child classes — don't confuse the two.
  • Runtime polymorphism requires both inheritance and method overriding to work.
  • Compile-time checks are based on the reference (declared) type; runtime behavior is based on the actual object type.
  • If a method isn't overridden in the actual object's class, Java searches up the inheritance chain (parent, grandparent, ... Object) for the nearest matching version.
