🧠 Key Concepts

Recap: The 4 OOPs Pillars and Their Benefits
  • Encapsulation — hides data; achieved via private fields + public getters/setters; gives security/control over data
  • Inheritance — reuses parent class functionality; saves time/money by not reinventing already-tested code
  • Polymorphism — achieves flexible, extensible behavior via method overriding; "open for extension, closed for modification"
  • Abstraction — hides implementation/logic; exposes only required functionality to other classes
"Open for Extension, Closed for Modification"

A well-designed system (using polymorphism) allows adding new functionality (e.g. a new payment type) without modifying existing, already-tested code. In real companies, modifying live production code that serves millions of users is extremely risky and costly to re-verify — so being able to "plug in" new behavior without touching existing classes is highly valued.

Abstraction — What It Means

While encapsulation hides data, abstraction hides implementation/logic. It means exposing only the necessary, required functionality to other classes, while keeping internal/core logic hidden. Analogy: a remote control exposes only buttons (required functionality) — the internal circuitry making the AC turn on/off (core logic) is completely hidden from the user.

How Abstraction is Implemented

Expose a single public method representing the required functionality (e.g. checkBalance()), while making all the internal step-by-step logic (connect to DB, validate credentials, decrypt, fetch balance, etc.) private methods that only the class itself can call. Other classes can only interact through the public method — they cannot bypass it to call the internal steps directly.

The Real Danger of NOT Following Abstraction

If internal/core methods are left public (not hidden), other classes might discover and start calling those internal methods directly, bypassing the intended public entry point. The developer of the original class may be unaware this is happening, since they assume everyone is going through the public method. If a fix or new feature (e.g. displaying a "monthly average balance not maintained" warning) is later added only inside the public method, callers who bypassed it directly will never receive that update — because they're not calling the code path where the fix was applied. This creates inconsistent, hard-to-debug behavior across the system.

Why This Matters More Than "Does It Work?"

Code that exposes internal methods might "work" fine initially — but it's a design flaw, not a functional one. The problem surfaces later, when changes/fixes need to be made and can't be guaranteed to reach every caller, because some callers found a way around the intended interface.

Garbage Collection — Unreachable Objects

An object becomes eligible for garbage collection when it becomes unreachable — meaning no active reference points to it anymore (e.g. after setting a reference variable to null, if nothing else refers to that same object). The garbage collector (GC) periodically scans for such unreachable objects and reclaims their memory.

System.gc() — Just a Request, No Guarantee

Calling System.gc() only requests garbage collection — it does not force immediate execution. The JVM decides when to actually run GC, often triggered automatically when it senses memory pressure (low available memory).

An Object Stays Reachable if ANY Reference Still Points to It

If multiple reference variables point to the same object, setting one of them to null does not make the object unreachable — as long as at least one other reference still points to it, the object remains reachable and is not eligible for garbage collection.

The Diamond Problem — Why Java Doesn't Allow Multiple Inheritance (Detailed)

If a class could extend two parent classes that both define a method with the same name but different implementations (e.g. both have method1(), but each returns something different), a child class inheriting from both would face genuine ambiguity — Java would have no way to determine which parent's version of the method to use. This structural conflict (two parents merging into one child) visually resembles a diamond shape, hence "Diamond Problem" or "Death of Diamond." This is the concrete reason multiple inheritance via classes is disallowed in Java (interfaces, covered separately, resolve this differently).

💻 Code Snippets / Examples

Abstraction — hiding core logic, exposing only checkBalance():

class PaymentProcessor {

    public int checkBalance(String userId, String password) {
        connectWithDB();
        getUserAndPasswordFromDB();
        validate();
        int balance = getBalanceFromDB();
        return balance;
    }

    private void connectWithDB() {
        System.out.println("Connecting with DB");
    }

    private void getUserAndPasswordFromDB() {
        System.out.println("Fetching user/password from DB");
    }

    private boolean validate() {
        return true;
    }

    private int getBalanceFromDB() {
        return 1200;
    }

}

Correct usage — other classes can only call the public entry point:

class User {
    public static void main(String[] args) {
        PaymentProcessor processor = new PaymentProcessor();
        int balance = processor.checkBalance("user1", "pass123"); // only this is callable
        // processor.connectWithDB(); -- NOT accessible, private
    }
}

Why bypassing abstraction causes problems (illustrative, not directly callable):

// If connectWithDB(), validate(), getBalanceFromDB() were PUBLIC instead of private,
// another class could call them directly, skipping checkBalance() entirely.
// Later, if a new warning message is added ONLY inside checkBalance():

public int checkBalance(String userId, String password) {
    connectWithDB();
    getUserAndPasswordFromDB();
    validate();
    int balance = getBalanceFromDB();

    if (balance < 2000) {
        System.out.println("Warning: monthly average balance not maintained");
    }

    return balance;
}
// Callers who bypassed checkBalance() and called getBalanceFromDB() directly
// will NEVER see this new warning — inconsistent behavior across the system.

Garbage collection — reachability example:

Payment p = new Payment(); // object created in heap, referenced by p

p = null; // object is now unreachable (if nothing else references it) -> eligible for GC

Payment p = new Payment();
Payment c = p; // c now also refers to the SAME object

p = null; // p no longer points to the object...
// but c STILL refers to it -> object remains REACHABLE, NOT eligible for GC

Diamond Problem — the exact ambiguity (conceptual, not valid Java):

class ParentA {
    public String method1() {
        return "Y"; // ParentA's version
    }
}

class ParentB {
    public String method1() {
        return "Z"; // ParentB's version — different implementation
    }
}

// NOT ALLOWED in Java:
// class Grandchild extends ParentA, ParentB {
//     // Ambiguity: which method1() should Grandchild inherit — "Y" or "Z"?
//     // Java has no way to resolve this -> multiple inheritance via classes is disallowed
// }

📌 Important Points / Common Mistakes

  • Abstraction hides implementation/logic (core, internal steps); encapsulation hides data (fields) — don't confuse the two pillars.
  • Always make internal step-by-step logic private, exposing only the intended public entry-point method.
  • A "working" system with exposed internals is still a design flaw — the real cost appears later when maintaining/updating the system.
  • An object is only eligible for garbage collection when it becomes fully unreachable — no remaining references anywhere.
  • System.gc() is a request, not a command — never rely on it for guaranteed immediate collection.
  • Java disallows multiple inheritance via classes specifically because of method-conflict ambiguity (the Diamond Problem) — interfaces solve this differently.
