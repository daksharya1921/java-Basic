🧠 Key Concepts

Recap: Inheritance = Code Reusability

Before writing new code, always check whether similar functionality already exists that can be reused via inheritance, rather than rebuilding from scratch — this saves significant development time.

Real-World Code Review Process

In companies, code doesn't go directly to production after being written. It typically goes through:

  • Static code review — automated tools (e.g. SonarQube, PMD, Checkstyle) that check things like unused variables, naming conventions, method/class length limits (e.g. no method over 200 lines), and other configurable rules.
  • Peer review (PR) — a senior developer or reviewer manually checks whether the code implements the business requirement correctly, and evaluates how it was implemented (structure, reusability, unnecessary duplication, proper use of local vs instance variables, etc.).
Why This Matters for Your Career

How well you apply reusability and good design (like inheritance) directly affects peer review feedback, which factors into performance reviews and appraisals/raises. Reviewers notice things like: could this have reused an existing class instead of taking days to rewrite similar logic? Following OOPs concepts properly (not just "knowing" them) leads to smoother career growth.

Private Members Are NOT Inherited

When a class extends a parent, it gets access to all non-private members (fields and methods) of the parent. Anything declared private in the parent remains inaccessible to the child class — private scope is strictly limited to the class where it's declared, regardless of inheritance.

Child Class Adds Its Own Requirements

A child class reuses common functionality from its parent and additionally implements whatever is specific to itself. Example: a general Account class has common operations (open, close, approve); a SavingAccount subclass reuses those and adds its own calculateInterest() method, since interest calculation is specific to savings accounts.

Single Inheritance

One class extends exactly one other class — a simple parent → child relationship.

Account → SavingAccount
Multi-Level Inheritance

A chain of inheritance spanning more than two levels — like a multi-story building. A grandchild class inherits from a child class, which itself inherits from a parent class.

Account → SavingAccount → PremiumSavingAccount
Hierarchical Inheritance

One parent class is extended by multiple different child classes, each independently. Example: both SavingAccount and CurrentAccount extend the same Account parent class.

Account
/   \
SavingAccount   CurrentAccount
Multiple Inheritance — NOT Supported in Java

Multiple inheritance means one class trying to extend more than one parent class directly. This is not supported in Java (unlike some other languages). This structure resembles a diamond shape and is famously known as the Diamond Problem (or "Death of Diamond" / DOD) — a classic interview question. The full reasoning behind why it causes issues is better understood once method overriding is covered.

ParentA   ParentB
  \     /
   Child
(NOT allowed in Java — a class cannot extend two classes)
The "IS-A" Test — Validating Inheritance Design

Before creating an inheritance relationship, check whether it forms a true, real-world "is-a" statement. Example: "Dentist is a Doctor" ✅ makes sense — valid inheritance. But a Cardiologist should not extend Dentist just because both are doctors — "Cardiologist is a Dentist" ❌ doesn't hold true in the real world. Both should instead independently extend the common parent, Doctor (hierarchical inheritance). Java will technically compile an illogical relationship, but it violates good design — the code should reflect reality.

💻 Code Snippets / Examples

Base Account class:

class Account {
    public int balance;
    public String accountHolderName;
    private String userId; // private — will NOT be inherited by children

    public void openAccount() {
        System.out.println("Account opened");
    }

    public void closeAccount() {
        System.out.println("Account closed");
    }

    public void approveAccount() {
        System.out.println("Account approved");
    }
}

SavingAccount — reuses parent + adds its own method:

class SavingAccount extends Account {

    public int calculateInterest(int amount) {
        double rate = 5; // 5%
        return (int) (amount * rate / 100);
    }

}
class Driver {
    public static void main(String[] args) {
        SavingAccount saving = new SavingAccount();
        saving.balance = 1000;              // inherited from Account
        saving.openAccount();               // inherited from Account
        saving.approveAccount();            // inherited from Account
        int interest = saving.calculateInterest(1000); // own method
        System.out.println(interest); // 50
        // saving.userId -- NOT accessible, private in parent
    }
}

Hierarchical inheritance — CurrentAccount also extends Account:

class CurrentAccount extends Account {

    public int calculateInterest(int amount) {
        return 0; // current accounts don't earn interest
    }

    public int forexCharges = 250; // flat charge per forex transaction, unlike savings which may be free

}

Multi-level inheritance — PremiumSavingAccount extends SavingAccount:

class PremiumSavingAccount extends SavingAccount {

    public void airportLoungeAccess() {
        System.out.println("Free airport lounge access");
    }

    public int movieDiscountPercent = 10;

}
class Driver {
    public static void main(String[] args) {
        PremiumSavingAccount premium = new PremiumSavingAccount();
        premium.openAccount();          // from Account (grandparent)
        premium.calculateInterest(1000); // from SavingAccount (parent)
        premium.airportLoungeAccess();   // own method
        // has access to fields/methods from ALL ancestors: Account + SavingAccount + itself
    }
}

Valid vs invalid "IS-A" inheritance design:

class Doctor { }

class Dentist extends Doctor { }     // valid — "Dentist is a Doctor" is true
class Cardiologist extends Doctor { } // valid — "Cardiologist is a Doctor" is true

// INVALID design (compiles, but logically wrong):
// class Cardiologist extends Dentist { }
// "Cardiologist is a Dentist" is NOT true in the real world

📌 Important Points / Common Mistakes

  • Private members are never inherited — they remain accessible only within their declaring class.
  • Always validate inheritance relationships against the real-world "is-a" test before implementing them.
  • Don't chain unrelated classes together just because they share a common ancestor — use hierarchical inheritance (both extending the same parent) instead of forcing an invalid parent-child relationship between siblings.
  • Java does not support multiple inheritance (a class extending two classes directly) — this is a deliberate design decision to avoid the Diamond Problem.
  • Code reusability and clean design directly impact real peer review feedback and performance evaluations in the industry.
