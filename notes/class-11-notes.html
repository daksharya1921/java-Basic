🧠 Key Concepts

Breaking a Task into Subtasks (Modular Code)

A large functionality (e.g. Fund Transfer) is made up of smaller subtasks — e.g. check receiver details, check sender details, check balance, then debit/credit. Instead of writing everything inside one giant method, each subtask should be implemented in its own method.

Why Modular Code is Better

If all logic is written inside a single method (e.g. thousands of lines), fixing one part (like balance checking) means digging through the entire method. With modular methods, you only need to open and fix the specific method responsible for that piece of logic — much easier to maintain.

Calling One Method from Another

A method is not limited to being called only from main — any method can call another method. Example: main calls fundTransfer(), and fundTransfer() in turn calls checkReceiverDetail(), checkSenderDetail(), and checkBalance().

Execution Flow Across Method Calls

When a method call statement is reached, control jumps to that method, executes its body, then returns control back to the exact line after the call — execution then continues from there. This happens in order, top to bottom, for each method call encountered.

Duplicate Variable Naming Issue

If a method already receives a value as a parameter (e.g. senderAccount), you should not re-declare a local variable with the same name inside that method — this creates a duplicate/conflicting declaration. The parameter itself already acts as a usable local variable inside the method.

Compilation vs Execution — Recap

The compiler only checks whether code is syntactically valid Java — it does not understand or care about business logic (e.g. whether fund transfer actually happens correctly). Code with correct syntax but a missing method call will compile successfully but produce no meaningful output during execution.

Meaningful Method Naming

Method names should clearly describe the action being performed — e.g. getUser(), findUser(), or createUser() instead of a vague name like user(), which could be mistaken for a variable or constant when reading the code.

Meaningful Comments

Writing a short one-line comment above a method describing its purpose (based on the requirement) helps keep the requirement in mind while coding and makes the code easier to review later.

💻 Code Snippets / Examples

Fund Transfer broken into subtask methods:

class PaymentProcessor {

    public static void main(String[] args) {
        String senderDetail = "SENDER123";
        String receiverDetail = "RECEIVER456";

        PaymentProcessor.fundTransfer(senderDetail, receiverDetail);
    }

    public static void fundTransfer(String senderAccount, String receiverAccount) {
        System.out.println("Fund transfer started");

        boolean receiverStatus = PaymentProcessor.checkReceiverDetail(receiverAccount);
        boolean senderStatus = PaymentProcessor.checkSenderDetail(senderAccount);
        int balance = PaymentProcessor.checkBalance(senderAccount);

        System.out.println("Fund transfer ended");
    }

    public static boolean checkReceiverDetail(String receiverAccount) {
        System.out.println("Checking receiver detail");
        return true;
    }

    public static boolean checkSenderDetail(String senderAccount) {
        System.out.println("Checking sender detail");
        return true;
    }

    public static int checkBalance(String senderAccount) {
        System.out.println("Checking balance");
        return 1000;
    }

}

Practice exercise — E-commerce place order (modular structure):

class EcomOrder {

    public static void main(String[] args) {
        String itemName = args[0];

        EcomOrder.placeOrder(itemName);
    }

    // Main task: place the order, which internally calls subtasks
    public static void placeOrder(String itemName) {
        System.out.println("Placing order for " + itemName);

        String invoice = EcomOrder.getInvoice(itemName);
        EcomOrder.sendOrderUpdateEmail(itemName);
        EcomOrder.payForItem(itemName);
    }

    public static String getInvoice(String itemName) {
        System.out.println("Generating invoice for " + itemName);
        return "INVOICE123";
    }

    public static void sendOrderUpdateEmail(String itemName) {
        System.out.println("Sending order update email for " + itemName);
    }

    public static void payForItem(String itemName) {
        System.out.println("Processing payment for " + itemName);
    }

}

📌 Important Points / Common Mistakes

  • A large task should be broken into smaller, well-named methods — one method per subtask.
  • Subtask methods should be called from the main task method (e.g. placeOrder), not directly from main.
  • Do not redeclare a parameter as a local variable with the same name inside the same method — this causes a duplicate declaration error.
  • Method calls transfer control to the called method, execute its body, then return control to the line right after the call.
  • Use command line arguments to keep programs flexible (many students hardcoded values instead of using args).
  • Give methods meaningful, action-based names (getInvoice, sendOrderUpdateEmail) instead of vague nouns.
  • Class names and file names must match, and both must follow proper capitalization (e.g. Business, not business).
  • Format code properly (consistent indentation/structure) — poorly formatted code is much harder to review.
