🧠 Key Concepts

Recap: Abstract Class = Partial Contract + Common Functionality

An abstract class combines two things: abstract methods (a contract — child classes must implement these) and concrete methods (shared functionality provided directly, usable by all children without reimplementation).

Why Force a Contract? Reason 1 — Preventing Naming Chaos

Without an enforced contract, if 20 different developers each independently implement "payment" functionality, you'd likely get 20 different method names (pay(), doPayment(), moneyTransfer(), sendMoney(), etc.) — all doing conceptually the same thing but named inconsistently. This makes a large system unmaintainable. An abstract class (or interface) forces everyone to use the same method name and signature, keeping the system uniform.

Why Force a Contract? Reason 2 — Centralized Control for Future Changes

Real requirements evolve — e.g. a new regulation requiring a transaction limit check might be introduced a year after launch. Without a central contract, you'd have to individually request every team/developer to add this — some might delay for months, some might never do it. With an abstract class/interface, you simply add a new abstract method to the contract — this immediately forces every implementing class to add the required functionality, or the code won't compile. This gives the system designer centralized control over the entire system's evolution.

Interfaces (Pre-Java 8) — 100% Contract, No Common Functionality

An interface is like an abstract class taken to its extreme: every method is abstract by default — there is no concrete/common functionality allowed at all (before Java 8). Use an interface when you want to specify only what must be done, with absolutely no shared implementation to provide.

Declaring an Interface 
interface IBanking {
    void deposit(double amount);
    void withdraw(double amount);
    void checkBalance();
}

Use the interface keyword instead of class. Every method inside is implicitly public and abstract, even without writing those keywords explicitly.

Only Abstract Methods and Constants Allowed

An interface can only contain: (1) abstract method declarations (no body), and (2) constants (implicitly public static final). Regular instance variables, constructors, and concrete (implemented) methods are not allowed in a pre-Java 8 interface.

interface ITicketBooking {
    int TICKET_PRICE = 500; // implicitly public static final — a constant

    void bookTicket();
    void cancelTicket();
}
No Constructors, No Object Creation

Interfaces have no constructors, and you cannot create an object of an interface — there's no state to initialize and no implementation to instantiate. An interface is purely a contract specification.

implements Keyword

A class doesn't extend an interface — it implements it:

class PersonalBanking implements IBanking {
    // must provide implementations for ALL methods declared in IBanking
}

Any class implementing an interface must provide a full implementation for every method declared in that interface — otherwise it won't compile.

Naming Convention: "I" Prefix

By industry convention, interface names are often prefixed with a capital I (e.g. IBanking, IUser) to visually distinguish them from regular class names at a glance. This is a convention, not a language requirement.

💻 Code Snippets / Examples

Abstract class recap — contract + common functionality:

abstract class PaymentSystem {
    public abstract void doPayment();
    public abstract void suspendPayment();
    public abstract void stopPayment();

    public void checkStatus() { // common functionality — shared by all children
        System.out.println("Checking payment status");
    }
}

Adding a new requirement forces ALL implementing classes to comply:

abstract class PaymentSystem {
    public abstract void doPayment();
    public abstract void suspendPayment();
    public abstract void stopPayment();
    public abstract void checkTransactionLimit(); // NEW requirement added a year later

    public void checkStatus() {
        System.out.println("Checking payment status");
    }
}
// Every existing subclass (UpiPayment, CardPayment, etc.) will now FAIL TO COMPILE
// until they implement checkTransactionLimit() — forcing system-wide compliance

Interface example — banking contract, 100% abstract:

interface IBanking {
    void deposit(double amount);
    void withdraw(double amount);
    void checkBalance();
    void transferFunds(double amount, String toAccount);
}
class PersonalBanking implements IBanking {
    @Override
    public void deposit(double amount) {
        System.out.println("Personal deposit: " + amount);
    }

    @Override
    public void withdraw(double amount) {
        System.out.println("Personal withdrawal: " + amount);
    }

    @Override
    public void checkBalance() {
        System.out.println("Checking personal balance");
    }

    @Override
    public void transferFunds(double amount, String toAccount) {
        System.out.println("Transferring " + amount + " to " + toAccount);
    }
}
class CorporateBanking implements IBanking {
    // must implement ALL four methods, with its own corporate-specific logic
    @Override
    public void deposit(double amount) { System.out.println("Corporate deposit"); }
    @Override
    public void withdraw(double amount) { System.out.println("Corporate withdrawal"); }
    @Override
    public void checkBalance() { System.out.println("Corporate balance check"); }
    @Override
    public void transferFunds(double amount, String toAccount) { System.out.println("Corporate transfer"); }
}

Constants in an interface:

interface ITicketBooking {

Why can't we just let each developer name their payment method however they want?It would create massive naming inconsistency across a large system, making it unmaintainable — a shared contract forces uniformity.
What's the practical benefit of forcing a contract, beyond naming consistency?Centralized control — when new requirements arise later, adding a method to the abstract class/interface immediately forces every implementing class to comply, rather than having to individually request each team.

📌 Important Points / Common Mistakes

  • All interface methods are implicitly public and abstract, even without writing those keywords.
  • Interfaces (pre-Java 8) only allow abstract method declarations and constants — no concrete methods, no instance variables, no constructors.
  • Classes use "implements" (not "extends") to signal they're fulfilling an interface's contract.
  • A class implementing an interface must provide implementations for every single method declared in it.
  • Prefixing interfaces with "I" (e.g. IBanking) is a naming convention to visually distinguish them from classes.
  • Interfaces cannot be instantiated — only concrete implementing classes can be.
