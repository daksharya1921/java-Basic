🧠 Key Concepts

Why We Write Methods

Code is written to perform tasks — e.g. placing an order, sending money, calculating a discount. Logic for a task should be written inside a method, not directly inside main. Even printing "Hello" happens inside a method (main is just one example of a method).

Components of a Method
  • Name — meaningful, camelCase (e.g. fundTransfer, sendEmail)
  • Return type — what the method sends back once its work is done
  • Access modifier — who can call/use the method: public, private, protected, or default (no modifier)
  • static (optional) — if present, the method belongs to the class and can be called using the class name, without creating an object
  • Parameters — input values supplied to the method to do its work
  • Body — the actual logic, enclosed in curly braces
Return Type

Once a method finishes its work, it can send a result back to whoever called it — this is the return type. Example: a "place order" method might return a String confirmation message with an order ID.

Parameters

Parameters are the inputs a method needs to do its job — similar to giving someone money and a quantity when asking them to "get coffee". A method can take zero, one, or multiple parameters, each with its own data type.

Declaring vs Calling a Method

Writing a method (declaring it) does not make it run. A method only executes when it is explicitly called by someone — either JVM (for main) or another method calling it. If nothing calls a method, its code never runs, even though the file compiles successfully.

Calling a Static Method

Since static methods belong to the class, they are called using: ClassName.methodName(arguments). This is the same reason JVM calls main as ClassName.main() — because main is static.

One Method Can Call Another

main doesn't have to do all the work itself — it can call other methods, and those methods can call further methods, similar to a contractor delegating different parts of a construction job to different vendors (painting, electrical, cement, etc.).

Why Store Results in Variables (Readability)

Storing a method's return value in a variable before printing it (rather than nesting everything in one line) makes code more readable and maintainable — important since code may remain in use for many years. The local variable used for this is short-lived and removed once the method finishes executing, so it doesn't cause memory concerns.

💻 Code Snippets / Examples

Example: placeOrder method (returns a String)

class Order {

    public static void main(String[] args) {
        String result = Order.placeOrder("iPhone 11", 5);
        System.out.println(result);
    }

    static String placeOrder(String itemName, int quantity) {
        System.out.println("Placing order for item " + itemName);
        return "Order has been placed, your order ID is AMJ12345";
    }

}

Example: addNumbers using command line arguments and a helper method

class AddNumbers {

    public static void main(String[] args) {
        System.out.println("Start");

        String input1 = args[0];
        String input2 = args[1];

        int number1 = Integer.parseInt(input1);
        int number2 = Integer.parseInt(input2);

        int result = AddNumbers.add(number1, number2);

        System.out.println("Sum of two numbers is " + result);

        System.out.println("End");
    }

    static int add(int firstNumber, int secondNumber) {
        int sum = firstNumber + secondNumber;
        return sum;
    }

}

class AddNumbers {

        System.out.println("End");
        // add() method exists elsewhere in the class but is never called here — it will not run
    }

        int sum = firstNumber + secondNumber;
        return sum;
    }

❓ Doubts / Questions Raised in Class

QuestionAnswer
Why did nothing happen when the placeOrder method was written but the program was run?Because the method was never called — writing a method doesn't execute it; something must explicitly call it.
Who calls the main method?JVM calls it, using ClassName.main(), because main is static.
Why use ClassName.methodName() to call a static method instead of just methodName()?Both work (the compiler adds the class name automatically), but explicitly writing the class name improves readability and clarity.
Does storing intermediate results in local variables (like sum) waste memory?No — local variables are removed once the method finishes executing; the readability benefit outweighs any negligible short-term memory use.

📌 Important Points / Common Mistakes

  • Avoid writing business logic directly inside the main method — write it inside separate, purposefully named methods.
  • A method must be called to execute; simply declaring it and compiling successfully does not run its code.
  • Static methods are called using ClassName.methodName(); this avoids needing to create an object (object creation not yet covered at this point).
  • Give methods meaningful, camelCase names describing what they do.
  • Always specify a proper return type — don't rely only on void when a method is expected to return a result.
  • Never assume — verify your understanding/approach (e.g. with a manager/lead) before investing significant time in an implementation.
