Class 21 — String via new Keyword, == vs equals(), and Object Identity

🎯 Topics Covered

  • Recap: String literal always goes to String Constant Pool (SCP), with deduplication
  • Creating Strings using new keyword — creates two objects (SCP + heap)
  • new keyword never checks for duplicates in the heap (unlike SCP)
  • Concatenation (CONCAT) result goes to the heap, not SCP
  • Reassignment required to actually use a modified/new String value
  • == operator — compares memory addresses (references)
  • equals() method — defined in the Object class, default behavior same as ==
  • String class overrides equals() to compare content instead of reference
  • Why Java allows both literal and new keyword — content vs identity use cases

🧠 Key Concepts

Recap: String Literal → Always SCP, With Deduplication

Anything written in double quotes is a literal and always results in (or reuses) an object in the String Constant Pool. Before creating a new one, Java checks whether an object with the exact same content already exists there — if so, it reuses it instead of creating a duplicate.

Creating a String with new — Creates TWO Objects 
String s1 = new String("Bangalore");

This line does two things:

  1. The literal "Bangalore" inside the parentheses is still a literal → one object is created (or reused) in the SCP.
  2. The new keyword then creates a separate, brand-new object in the Heap, using that content — and s1 refers to this heap object, not the SCP one.

So a single new String("...") statement typically creates 2 objects total (assuming the SCP one didn't already exist).

new Keyword Never Deduplicates in the Heap

Unlike the SCP, object creation via new in the heap never checks whether an identical object already exists — it always creates a brand-new object, every single time, regardless of duplicate content elsewhere in the heap.

Concatenation (CONCAT) Result Goes to the Heap

When you modify a String (e.g. using .concat()), the resulting new object is placed in the heap, not the SCP — even though the original string may have come from the SCP. The original object in the SCP remains unchanged (immutability).

Reassignment Needed to Use the New Value

If you call a method like .concat() but don't reassign the result back to a variable, nothing refers to the newly created object — the original reference still points to the old, unmodified value.

== Operator — Compares Memory Addresses

== checks whether two references point to the exact same object in memory (same address) — not whether their content matches. Always draw out the memory diagram (SCP/heap, addresses) when unsure, rather than guessing based on displayed values.

equals() — Defined in the Object Class

The equals() method exists in the Object class — the universal parent of every Java class. By default, equals() behaves exactly like ==: it compares memory addresses/references, not content. Every class inherits this default behavior unless it explicitly changes it.

String Class Overrides equals() to Compare Content

The String class overrides (changes) the default equals() behavior inherited from Object — instead of comparing memory addresses, String's equals() compares the actual content/characters of the two strings. This is why "abc".equals("abc") returns true even if they're different objects with different addresses.

equals() Behavior is Not Universal — It Depends on the Class

Only classes that specifically override equals() (like String) compare content. For a plain custom class (e.g. Employee) with no overridden equals(), calling .equals() falls back to the Object class's default — comparing memory addresses, just like ==.

Why Both Literal and new Exist — Content vs Identity

If Java only allowed literals, you could never have multiple distinct objects with identical content — but some real use cases genuinely need this. Example: tracking the same value at different points/locations (identity matters more than content) — e.g. representing the same person's data as they move between locations, where each snapshot needs to be a separate, distinct object even though the content might be the same. The new keyword supports this "identity matters" use case; literals support the more common "content/value matters, avoid duplication" use case.

💻 Code Snippets / Examples

String via new keyword — creates 2 objects:

String s1 = new String("Bangalore");
// Object 1: "Bangalore" created/reused in SCP (from the literal argument)
// Object 2: new "Bangalore" object created in the Heap
// s1 refers to the HEAP object, not the SCP one

new never deduplicates — repeated calls always create new heap objects:

String s10 = "Mumbai";           // SCP object created (1st time)
String s11 = "Mumbai";           // reuses same SCP object
System.out.println(s10 == s11);  // true — same SCP object

String s13 = new String("Mumbai"); // SCP: "Mumbai" already exists, reused
                                    // Heap: brand new object created
String s14 = new String("Mumbai"); // SCP: reused again
                                    // Heap: ANOTHER brand new object created
System.out.println(s13 == s14);  // false — two different heap objects

Concatenation — result goes to heap, reassignment required:

String city = "Bangalore"; // SCP object
city = city.concat(" India"); // new object "Bangalore India" created in HEAP
                               // city is REASSIGNED to point to it
System.out.println(city); // Bangalore India

String city2 = "Delhi";   // SCP object
city2.concat(" India");   // new object created in heap, but NOT reassigned
System.out.println(city2); // still prints "Delhi" — original SCP object unchanged

equals() default behavior (Object class) vs overridden (String class):

class Employee { }

Employee e1 = new Employee();
Employee e2 = new Employee();

System.out.println(e1 == e2);        // false — different heap addresses
System.out.println(e1.equals(e2));   // false — Employee doesn't override equals(),
                                      // so it uses Object's default (same as ==)
String s1 = new String("Java");
String s2 = new String("Java");

System.out.println(s1 == s2);        // false — different heap objects (new keyword)
System.out.println(s1.equals(s2));   // true — String overrides equals() to compare content

📌 Important Points / Common Mistakes

  • String literal → SCP only, with deduplication. new String(...) → SCP (for the literal argument) + a separate, always-new heap object.
  • The heap never deduplicates on new — every new call creates a fresh object.
  • == always compares addresses/references — never content — regardless of class.
  • equals() defaults to == behavior (Object class) unless a class specifically overrides it (like String does).
  • Don't blindly assume equals() always compares content — that's specific to classes (like String) that override it, not a universal Java rule.
  • Concatenation results go to the heap; without reassignment, the original variable still points to the unmodified original object.
  • Draw memory diagrams (SCP + heap boxes with addresses) when working through String object-counting problems — don't just guess from printed output.
