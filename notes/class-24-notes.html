🧠 Key Concepts

When to Use String vs StringBuilder

Use plain String when the value is static/fixed (e.g. a fixed message or label that won't be modified). Use StringBuilder/StringBuffer when content needs to be dynamically constructed or modified repeatedly — e.g. templates where placeholders (name, order status, date) change per user. Using String for frequently-modified content would create a huge number of throwaway objects (since Strings are immutable), risking memory issues at scale.

Real-World Use Case: Dynamic Notification Templates

Example: an e-commerce order confirmation email — "Hi [name], thank you for placing your order for [item]. Your order is currently [status]. You will receive it by [date]." This same template structure is reused for every customer, with different values substituted each time. Using StringBuilder, the same object is appended/modified per request instead of creating a brand-new String object combination every time — critical when generating potentially millions of such messages.

StringBuilder.append() — No New Object Per Modification

Calling .append() on a StringBuilder modifies the existing object in place — it does not create a new object each time, unlike String concatenation. This is the core reason StringBuilder is preferred for repeated/dynamic text construction.

Constructors Should Only Initialize State

Business logic (e.g. constructing a SQL query, sending a notification) should generally not be written inside a constructor — a constructor's job is to initialize the object's fields. Business logic belongs in separate, purpose-named methods, unless there's a specific requirement forcing otherwise.

Practice: Dynamic Query Builder

Exercise: build a SQL query string dynamically based on which inputs are provided:

  • No input → SELECT * FROM employee (get all records)
  • Employee ID provided → SELECT * FROM employee WHERE id = <value>
  • Salary provided (no ID) → SELECT * FROM employee WHERE salary > <value>

This demonstrates conditionally building a String/query using StringBuilder based on runtime input, rather than hardcoding a single fixed query.

Software Design Principle #1: Maintainability

Code in real companies typically lives for 10–15+ years. Just like a physical product needs to be repairable/serviceable over its lifetime, code needs to be maintainable — meaning other developers (including ones who didn't originally write it) should be able to understand it, fix issues, and extend it without excessive difficulty.

Software Design Principle #2: Reusability

Code should not be built for a single, overly narrow use case only. Example: an email template built for "order placed" notifications should ideally also be usable (with minor changes) for "order cancelled" or "payment failed" notifications, rather than requiring a completely separate implementation for every slightly different scenario. Writing reusable code saves significant development and testing time on future, similar requirements.

Software Design Principle #3: Modularity

Large applications (real systems may have 100,000+ classes) should be organized into logically separated modules (e.g. payment module, cart module, search module) — similar to how a house has separate rooms for different purposes (kitchen, bedroom, study). This way, if an issue is reported in one area (e.g. "UPI payment not working"), developers can focus only on the relevant module, without the risk of unrelated systems being impacted, and without having to search through unrelated code.

OOP as the Foundation for Application Design

Object-Oriented Programming (OOP) is the foundation for designing applications that are maintainable, reusable, and modular — using objects as the central building blocks. How well an application achieves these three qualities largely depends on how well OOP principles are applied. This connects to Low-Level Design (LLD), which is heavily based on OOP concepts, and to design principles like SOLID (mentioned as an upcoming topic).

💻 Code Snippets / Examples

Dynamic notification email using StringBuilder:

class EmailTemplate {
    String name;
    String item;
    String status;
    String deliveryDate;

    public EmailTemplate(String name, String item, String status, String deliveryDate) {
        this.name = name;
        this.item = item;
        this.status = status;
        this.deliveryDate = deliveryDate;
    }
}
class OrderNotification {

    public String sendNotification(EmailTemplate emailMessage) {
        StringBuilder finalMessage = new StringBuilder();

        finalMessage.append("Dear ")
                    .append(emailMessage.name)
                    .append(", thank you for placing an order for item ")
                    .append(emailMessage.item)
                    .append(". Your order status is ")
                    .append(emailMessage.status)
                    .append(". You will receive it by ")
                    .append(emailMessage.deliveryDate)
                    .append(".");

        return finalMessage.toString();
    }

}
class Driver {
    public static void main(String[] args) {
        EmailTemplate msg1 = new EmailTemplate("Krishna", "Laptop", "Packed", "10 July");
        OrderNotification notifier = new OrderNotification();
        String email = notifier.sendNotification(msg1);
        System.out.println(email);
    }
}

Practice exercise — dynamic SQL query builder:

class QueryConstructor {

    public String constructQuery(String employeeId, String salary) {
        StringBuilder query = new StringBuilder("SELECT * FROM employee");

        if (employeeId != null) {
            query.append(" WHERE id = ").append(employeeId);
        } else if (salary != null) {
            query.append(" WHERE salary > ").append(salary);
        }

        return query.toString();
    }

}
class Driver {
    public static void main(String[] args) {
        QueryConstructor qc = new QueryConstructor();

        System.out.println(qc.constructQuery(null, null));       // SELECT * FROM employee
        System.out.println(qc.constructQuery("101", null));      // SELECT * FROM employee WHERE id = 101
        System.out.println(qc.constructQuery(null, "50000"));    // SELECT * FROM employee WHERE salary > 50000
    }
}

📌 Important Points / Common Mistakes

  • Prefer StringBuilder/StringBuffer over String whenever content needs to change dynamically or repeatedly.
  • Keep constructors focused purely on field initialization — avoid embedding business logic there.
  • Design code with maintainability, reusability, and modularity in mind from the start, not as an afterthought.
  • No software is ever completely bug-free — design so that fixes can be made with minimal, contained changes rather than large rewrites.
  • Real production systems can have hundreds of thousands of classes — modular organization is essential at that scale.
