🧠 Key Concepts

Constructor Chaining — The Full Flow
new keyword → this class's constructor → super() (parent constructor) → ... → Object class constructor

When you create an object with new, it calls that class's constructor. If the class has a superclass, that constructor calls the superclass's constructor, and so on, until it reaches the built-in Object class constructor at the top. This chain — whether direct or indirect — is called constructor chaining.

super vs this in Constructors
  • super(...) — calls the parent class's constructor
  • this(...) — calls another constructor in the same class (constructor overloading)

Whichever is used, it must be the first line of the constructor. The chain must eventually reach the Object class, directly or indirectly (e.g. class A's constructor calls super leading to B, which calls super leading to C, which eventually reaches Object).

Real-World Use Case: Guest User Default Constructor (this())

Example: on an e-commerce site, if you browse and add items to cart without logging in, the system still tracks you via a guest user object, often using a browser cookie ID as the only known value. Since the caller (e.g. a webpage) doesn't know the user's real name, email, or mobile number, it calls a simple constructor with just the cookie/user ID. That constructor internally calls a fuller constructor (using this(...)) supplying default placeholder values (e.g. "Guest User") for the fields it doesn't have — so the object still gets created consistently with default data.

Why Not Call the Full Constructor Directly?

The caller (e.g. main/driver class, or a webpage) may only have access to one piece of information (like a cookie ID) — it has no way to supply values it doesn't have. So the simpler constructor is the only valid option for that caller; internally, that constructor delegates to the fuller one using this(...) with generated default values.

Instance vs Static — Practical Example

Instance variable example: quantity in a specific user's cart — specific to that user/object. Static variable example: a coupon code like "MMT100" — shared/common across all objects, one copy for the entire class.

Static Initialization Block (SIB) 
static {
    // code here
}

Executes only once — right after the class is loaded into memory, before the main method or any constructor runs. Used for setup that's needed once for the entire class, regardless of how many objects are created — e.g. opening a database connection or file, at program startup.

Instance Initialization Block (IIB) 
{
    // code here
}

Executes every time an object is created — runs as if it were inserted as the "second line" of every constructor (right after the mandatory super/this call), before the rest of the constructor body executes. Used for setup/preconditions needed for every individual object.

Order of Execution
Class loads → SIB runs (once) → main() starts → for each object: super/this call → IIB runs → rest of constructor body runs

If a class has multiple IIBs, they execute top to bottom, and this full set runs once per object created (e.g. 3 IIBs × 5 objects = 15 total IIB executions).

Practical Use Case: Counting Live Viewers/Objects

Platforms like Hotstar/YouTube track "live viewers" essentially by counting objects created — each time a user logs in/connects, a user-session object is created, incrementing a shared static counter. This static counter can be incremented inside an IIB (which runs for every object/user created), giving a running total of active sessions/objects.

💻 Code Snippets / Examples

Constructor chaining using this() — guest user pattern:

package com.amazon;

class User {

    private String username;
    private String userId;
    private String mobile;

    // Full constructor — used when complete details are available
    public User(String username, String userId, String mobile) {
        this.username = username;
        this.userId = userId;
        this.mobile = mobile;
    }

    // Guest constructor — only cookie/user ID known, delegates to full constructor
    public User(String userId) {
        this("Guest User", userId, "Not Provided");
    }

}
class Driver {

    public static void main(String[] args) {
        User guest = new User("cookieId123"); // caller only has the cookie ID
    }

}

Static Initialization Block (SIB) and Instance Initialization Block (IIB):

class Demo {

    static {
        System.out.println("Static block"); // runs once, when class loads
    }

    {
        System.out.println("Instance block"); // runs every time an object is created
    }

    public Demo() {
        System.out.println("Constructor");
    }

    public static void main(String[] args) {
        System.out.println("Main method start");
        Demo d1 = new Demo();
        Demo d2 = new Demo();
    }

}

// Output order:
// Static block
// Main method start
// Instance block
// Constructor
// Instance block
// Constructor

Practical example — counting objects created (live viewer style tracking):

class Viewer {

    public static int totalViewers = 0;

    {
        totalViewers = totalViewers + 1; // runs for every object created
    }

    public Viewer() {
        // default/free viewer
    }

    public Viewer(String type) {
        // paid viewer, type supplied
    }

}
class Driver {

    public static void main(String[] args) {
        Viewer v1 = new Viewer();
        Viewer v2 = new Viewer("Paid");
        Viewer v3 = new Viewer();

        System.out.println("Total viewers: " + Viewer.totalViewers); // 3
    }

}

📌 Important Points / Common Mistakes

  • super() or this() must always be the first line inside a constructor.
  • The constructor chain must ultimately reach the Object class constructor, directly or indirectly.
  • SIB executes once per class load; IIB executes once per object creation — don't confuse the two.
  • Use SIB for class-wide setup (e.g. DB/file connections); use IIB for per-object setup/preconditions.
  • To access a static member, use ClassName.memberName; to access a non-static (instance) member, you need an object reference.
