🧠 Key Concepts

Access Modifiers — Quick Recap

Access modifiers control visibility — i.e. which other classes can access a given method/variable. Example: a method that saves sensitive card details (e.g. inside a CreditCardProcess class) should be private so no other class can call it directly, for security reasons.

What is Control Flow?

Control flow lets a program make decisions based on conditions — every application constantly evaluates conditions (e.g. "if order value ≥ 350, free delivery"; "if marks > 60%, allow placement"). Without control flow, no meaningful application logic can be written.

Condition vs Business Logic

In an if statement, the part inside parentheses is the condition being checked; the code inside the block that runs when the condition is true is the business logic.

Unreachable Code After Return

Once a method hits a return statement, execution exits the method immediately. Any code written after the return statement (in the same method, same path) is unreachable and causes a compiler error: unreachable statement.

Handling Multiple Conditions — else if

A simple if-else handles only two outcomes. For more than two possible outcomes, use a chain of if / else if / else if / ... / else. At runtime, conditions are checked top to bottom; as soon as one condition is true, its block executes and all remaining conditions are skipped — only one block ever executes per run.

else as the Default/Fallback

The final else (with no condition) acts as a fallback — it executes only if none of the preceding if/else-if conditions were true.

Boundary (Corner) Cases

Boundary cases are edge values exactly at a threshold (e.g. exactly ₹10,000 or exactly ₹5,000). Using > instead of >= (or < instead of <=) can incorrectly exclude these edge values from the intended range. Always check the requirement carefully to decide whether boundary values should be included.

Always Use Curly Braces

Even for a single-statement if/else body, always wrap it in curly braces { }. Without braces, only the very next line is treated as part of the conditional block — any additional line added later (e.g. to also send an email) will run unconditionally, regardless of the condition, which is a common and dangerous bug.

Developer Role vs User Role

While writing and compiling code, you are the "developer." Once you run the program and supply input, you become the "user" supplying real input — this is a useful mental shift when testing your own programs (e.g. via command line arguments).

💻 Code Snippets / Examples

Single condition — checking delivery eligibility:

class ControlFlow {

    public static void main(String[] args) {
        int amount = Integer.parseInt(args[0]);
        boolean result = ControlFlow.checkFreeDelivery(amount);
        System.out.println("Customer delivery status: " + result);
    }

    private static boolean checkFreeDelivery(int amount) {
        boolean status = false;

        if (amount >= 10000) {
            status = true;
        } else {
            status = false;
        }

        return status;
    }

}

Multiple conditions — travel decision using if / else-if / else:

class TravelDecision {

    public static void main(String[] args) {
        int amount = Integer.parseInt(args[0]);
        TravelDecision.takeDecision(amount);
    }

    private static void takeDecision(int amount) {
        if (amount >= 10000) {
            System.out.println("User will go by flight");
        } else if (amount < 10000 && amount >= 5000) {
            System.out.println("User will go by train");
        } else if (amount < 5000 && amount >= 1000) {
            System.out.println("User will go by bus");
        } else {
            System.out.println("User does not feel like going");
        }
    }

}

Practice exercise — income category classification:

package com.codewala;

class IncomeCategory {

    public static void main(String[] args) {
        int yearlyIncome = Integer.parseInt(args[0]);
        IncomeCategory.checkIncomeCategory(yearlyIncome);
    }

    private static void checkIncomeCategory(int yearlyIncome) {
        if (yearlyIncome < 200000) {
            System.out.println("Category: BPL");
        } else if (yearlyIncome >= 200000 && yearlyIncome < 1000000) {
            System.out.println("Category: Middle Class");
        } else if (yearlyIncome >= 1000000 && yearlyIncome < 10000000) {
            System.out.println("Category: Upper Middle Class");
        } else {
            System.out.println("Category: High Net-worth Individual");
        }
    }

}

Why curly braces matter — bug example:

// WITHOUT braces — dangerous:
if (amount >= 10000)
    System.out.println("Eligible");
    sendEmail(); // ⚠ this line ALWAYS runs, regardless of the condition!

// WITH braces — correct:
if (amount >= 10000) {
    System.out.println("Eligible");
    sendEmail(); // only runs when condition is true
}

❓ Doubts / Questions Raised in Class

QuestionAnswer
If none of the if/else-if conditions are true, what happens?The final else block executes, acting as a default/fallback.
Do all conditions get checked even after one is found true?No — as soon as one condition evaluates true, its block runs and all remaining conditions are skipped entirely.
What happens if a boundary value (e.g. exactly 10,000) isn't handled with >= or <=?It gets excluded from the intended range — always verify boundary conditions match the actual requirement.
Why did the program throw an "array index out of bounds" error?No command line argument was supplied when running the program, so args[0] didn't exist — this is a runtime exception, not a compile-time error.

📌 Important Points / Common Mistakes

  • Only one block in an if/else-if/else chain executes per run — never assume multiple blocks run together.
  • Always double-check boundary values (>= vs >, <= vs <) against the actual requirement.
  • Always use curly braces for if/else bodies, even single-line ones, to avoid silent logic bugs later.
  • Give classes and methods names that accurately reflect their purpose (e.g. a class named "JomatoOrder" should not contain unrelated logic like checking marks).
  • Use packages for all classes going forward, as covered in the previous class.
  • Method return types must match what's actually being returned — e.g. don't return double/int when the output is meant to be a category label (String).
  • Business logic belongs inside methods, called from main — not written directly inside main.
