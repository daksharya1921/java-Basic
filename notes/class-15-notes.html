🧠 Key Concepts

Where Eclipse Stores Your Code

Eclipse is just an IDE (tool) to help develop code — the actual source files are stored on disk under the project's workspace folder (Project → Properties → Resources shows the exact path), inside the package folder structure (e.g. src/com/codewala/).

Why Debugging Matters

In real company environments, nobody walks you through every class — you're often just told "check this class." Debugging lets you pause execution and inspect the internal state (variable values) of a running program at a specific point, which is essential for understanding unfamiliar code.

Breakpoints

A breakpoint marks a line where execution should pause when running in debug mode. Without setting a breakpoint first, debug mode will simply run through the program like a normal run — you must place the breakpoint on the line where you want control to stop.

Debug Navigation Shortcuts
F6 — Step Over (next line) F5 — Step Into (go inside a method) F7 — Step Return (go back to caller) F8 — Resume (jump to next breakpoint)
  • F6 — moves to the next line in the current method (step over)
  • F5 — steps inside the method being called on the current line
  • F7 — returns control back to where the current method was called from
  • F8 — resumes execution until the next breakpoint is hit, skipping line-by-line stepping
Terminate Debug Sessions Properly

Each debug run starts a new JVM process that occupies memory. If previous debug sessions aren't terminated, memory builds up and can eventually prevent new sessions from launching, or slow down the system. Always terminate a debug session when done before starting a new one.

switch-case — What It Replaces

switch is a cleaner alternative to writing a long chain of if / else if / else if / ... statements when checking multiple possible values of a single variable/expression.

switch-case Syntax 
switch (variable) {
    case value1:
        // logic
        break;
    case value2:
        // logic
        break;
    default:
        // logic if nothing matched
}
Why break is Required in Each Case

Without break, execution falls through to the next case(s) even after a match is found, continuing until it hits a break or reaches the end of the switch block. Always add break at the end of each case (as best practice) to avoid unintended fall-through bugs — this is a very common exam/interview question.

default Case

The default case executes when none of the specified case values match. It doesn't have to be written last (though that's the conventional placement) and doesn't strictly require its own break if it's the final block.

Printing vs Returning — Why It Matters

If a method just prints a result (System.out.println) instead of returning it, the calling class/method has no way to access that result — printed output only appears in the console of the class that printed it, and is invisible to other classes. For another class to use a method's result, that method must return the value (not just print it). Example: a payment app's balance-check method must return the balance, not just print it — otherwise the app UI would never show the balance.

Calling Methods Across Multiple Classes

Not every class needs a main method — typically only one class has main, and other classes contain reusable logic exposed through public methods, called from the main class (or from each other).

Code Refactoring

Refactoring means reworking existing, working code to remove duplication or improve structure — without changing its functional behavior. Example: if two nearly identical methods exist for handling similar cases (e.g. separate methods for retail vs agent ticket booking), they can often be merged into a single method with proper condition-checking logic.

💻 Code Snippets / Examples

switch-case — identify day from a number:

package com.codewala;

class SwitchDemo {

    public static void main(String[] args) {
        int number = Integer.parseInt(args[0]);
        String result = SwitchDemo.identifyDay(number);
        System.out.println(result);
    }

    public static String identifyDay(int number) {
        String day = null;

        switch (number) {
            case 1:
                day = "Monday";
                break;
            case 2:
                day = "Tuesday";
                break;
            case 3:
                day = "Wednesday";
                break;
            default:
                day = "Unknown";
        }

        return day;
    }

}

Why printing instead of returning breaks caller usage (bad example):

// BAD: caller cannot access this result
public static void identifyDay(int number) {
    String day = "Wednesday";
    System.out.println(day); // only visible in THIS class's console output
}

// GOOD: caller receives the result
public static String identifyDay(int number) {
    String day = "Wednesday";
    return day; // caller can now use this value
}

switch-case — discount calculator by customer type:

private static void calculateDiscount(String type) {
    switch (type) {
        case "Silver":
            // silver customer discount logic
            break;
        case "Gold":
            // gold customer discount logic
            break;
        default:
            // non-standard/unknown customer type
    }

Practice exercise — IRCTC-style ticket booking across two classes:

// File: IRCTC.java
package com.codewala;

class IRCTC {

    public static void main(String[] args) {
        Booking.bookTicket("Retail", 3);
    }

}
// File: Booking.java
package com.codewala;

class Booking {

    public static void bookTicket(String userType, int numberOfTickets) {
        Booking.validateBooking(userType, numberOfTickets);
    }

    private static void validateBooking(String userType, int numberOfTickets) {
        if (userType.equals("Retail")) {
            if (numberOfTickets > 6) {
                System.out.println("Not allowed, retail user can book max 6 tickets");
            } else {
                System.out.println("Booking confirmed for retail user");
            }
        } else if (userType.equals("Agent")) {
            System.out.println("Booking confirmed for agent, any number of tickets allowed");
        } else {
            System.out.println("Booking is not supported for unknown user type");
        }
    }

}

❓ Doubts / Questions Raised in Class

QuestionAnswer
Why does debug mode just run through without pausing?A breakpoint must be set first — without one, debug mode behaves like a normal run.
Why isn't a printed value inside a method visible to the class that called it?Console output is only visible to the class/process that printed it; other classes need the value to be returned, not printed.
Is break mandatory in every switch case?Not syntactically mandatory, but functionally essential — without it, execution falls through into subsequent cases, causing incorrect behavior. Treated as required best practice.
Does every class need a main method?No — only one class typically has main; other classes provide reusable methods called from main or from each other.
Why refactor code that already produces correct output?To remove duplicate logic and improve maintainability, even if the output is already correct — this is standard practice in real development.

📌 Important Points / Common Mistakes

  • Always terminate debug sessions after use to avoid memory buildup from multiple running JVM processes.
  • Always include break at the end of each switch case unless intentional fall-through is required.
  • A method meant to be used by other classes must return its result — printing alone is insufficient.
  • Avoid duplicating similar logic across multiple methods; refactor into a single method with proper conditions where possible.
  • Close class and method braces immediately as you write them to avoid confusing "missing token" compiler errors later.
  • Focus on solving the business problem correctly; the specific Java version/newest features are secondary to solid fundamentals.
