🧠 Key Concepts

Recap: Why Encapsulation Matters

Encapsulation isn't just "hiding for the sake of hiding" — it prevents data discrepancy caused by uncontrolled modification from other classes. When reviewing code (e.g. in interviews), the correct evaluation lens is "does this follow OOPs principles," not just "does it compile and run."

What is Inheritance? (Real-World Analogy)

Just like a child automatically inherits their parent's property/wealth without starting from zero, a class can inherit the properties and functionality of another class. Example: if a parent has 100 acres of land and the child adds 10 more acres through their own effort, the child ends up with 110 acres total — they didn't have to "earn" the original 100 acres from scratch.

Why Inheritance Exists — Code Reusability

If functionality already exists in a class (e.g. a notification-sending class running stably in production for years, already tested and bug-fixed), rewriting similar functionality from scratch for a new class wastes significant time and reintroduces risk (new bugs, re-testing, re-approval cycles). Inheritance lets a new class reuse that existing, proven functionality instead.

extends Keyword — Establishing the Relationship 
class ResellerNotification extends Notification {
    // automatically gets access to Notification's members
}

Using extends makes the class on the left the child and the class on the right the parent. All functionality declared in the parent becomes available in the child without rewriting it.

Superclass and Subclass Terminology
  • Superclass (or parent class) — the class being inherited from
  • Subclass (or child class) — the class that inherits, using extends
Object Class — The Universal Root Parent

Recap/reinforcement: every class in Java is, directly or indirectly, a child of the built-in Object class. This is why methods like hashCode() are callable on any object even if never explicitly written in that class or its immediate parent — they're inherited all the way from Object, at the top of every inheritance chain.

Multi-Level Inheritance

Inheritance chains can go several levels deep — a class can have a parent, which itself has a parent, and so on. A child class at the bottom of such a chain has access to properties from all its ancestors (parent, grandparent, etc.), not just its immediate parent.

"IS-A" Relationship

Inheritance models an "is-a" relationship: an Electronic Product is a type of Product; Men's Clothing is a type of Clothing Product, which itself is a type of Product. If this natural "is-a" relationship holds true, inheritance is the appropriate design choice.

Child Class Can Add Its Own Properties

A subclass isn't limited to only the inherited members — it can declare additional fields and methods specific to itself. Example: a general Product class might have id/name/price, while ElectronicProduct (its child) additionally has a warrantyYears field and a warrantyDetails() method that general products don't have.

💻 Code Snippets / Examples

Reusing existing notification functionality via inheritance:

package com.amazon.retail.notification;

class Notification {
    public void sendNotification(String emailAddress, String message) {
        // existing, tested logic — e.g. connects to mail server and sends email
        System.out.println("Sending email to " + emailAddress + ": " + message);
    }
}
class ResellerNotification extends Notification {
    // no need to rewrite sendNotification — inherited from Notification
}
class Driver {
    public static void main(String[] args) {
        ResellerNotification resellerNotification = new ResellerNotification();
        resellerNotification.sendNotification("reseller@example.com", "Stock is low, please refill");
        // reusing parent class functionality — nothing new written here
    }
}

Simple example — reusing fields and a method:

class UserManagement {
    String type;
    String location;

    void doSomething() {
        System.out.println("Doing something in UserManagement");
    }
}
class User extends UserManagement {
    // no new members — everything reused from parent
}
class Driver {
    public static void main(String[] args) {
        User user = new User();
        user.doSomething();      // inherited method, works even though User never wrote it
        System.out.println(user.type);     // inherited field
        System.out.println(user.location); // inherited field
    }
}

Product hierarchy — Electronic Product adds its own property:

class Product {
    public String productId;
    public String productName;
    public int price;

    public void productDetails() {
        System.out.println(productId + " " + productName + " " + price);
    }
}
class ElectronicProduct extends Product {
    public int warrantyYears = 2; // additional property, specific to electronic products

    public void warrantyDetails() {
        System.out.println("Warranty: " + warrantyYears + " years");
    }
}
class Driver {
    public static void main(String[] args) {
        ElectronicProduct ep = new ElectronicProduct();
        ep.productId = "P1";
        ep.productName = "Laptop";
        ep.price = 50000;

        ep.productDetails();   // inherited from Product
        ep.warrantyDetails();  // defined in ElectronicProduct itself
        // total: 3 inherited fields + 1 own field = 4 fields available in ElectronicProduct
    }
}

❓ Doubts / Questions Raised in Class

QuestionAnswer
Why not just write new functionality from scratch instead of using inheritance?Because reusing existing, already-tested, production-proven code saves significant time and avoids reintroducing bugs that come with a completely new implementation.
How is a class able to call methods (like hashCode()) it never wrote?Through inheritance — every class is implicitly a child of the Object class, which provides such methods to all classes.
What properties does a class at the bottom of a multi-level hierarchy have access to?Its own declared properties, plus everything inherited from its immediate parent, plus everything from all ancestors further up the chain (grandparent, etc.).
How do you decide if inheritance is the right relationship to use?Check whether an "is-a" relationship genuinely holds — e.g. "Electronic Product is a Product" makes sense, so inheritance fits.

📌 Important Points / Common Mistakes

  • Use extends to create a parent-child (superclass-subclass) relationship between classes.
  • Before writing new functionality, check whether similar functionality already exists and can be reused via inheritance.
  • A subclass can both reuse inherited members and add its own new fields/methods.
  • Always verify an "is-a" relationship makes logical sense before applying inheritance — don't force inheritance where it doesn't naturally fit.
  • Remember to save files (Ctrl+S) before compiling — several students in class lost time due to unsaved changes.
