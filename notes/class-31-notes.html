🧠 Key Concepts

Two Types of Polymorphism
Runtime PolymorphismCompile-time Polymorphism
Achieved via Method OverridingAchieved via Method Overloading
Decision of which method to call happens at runtime, based on the actual object typeDecision of which method to call happens at compile time, based on the method signature used in the call
Why It's Called "Runtime" vs "Compile-time"

In overriding, the compiler doesn't know in advance which overridden version will run — that's determined only once the program executes and the actual object is known. In overloading, the compiler can determine exactly which version will be called just by looking at the argument types/count/order at compile time — no runtime ambiguity exists.

The Real Advantage of Polymorphism: Extensibility Without Modification

The core practical value of polymorphism isn't just "calling different methods" — it's designing a system so that new functionality can be added without touching existing, already-tested, already-deployed code. This matters enormously in real production systems, where modifying live code requires expensive re-testing and re-verification cycles, and where the system can't simply be "paused" while changes are made (since real users depend on it continuously).

The "Processor/Router" Pattern

A common design: build a generic method (e.g. processPayment(Payment payment)) that accepts the parent type as its parameter. Internally, it simply calls the (possibly overridden) method on that parameter — like payment.doPayment(). Because of runtime polymorphism, this single generic method automatically routes to the correct specific implementation, no matter which child object is actually passed in.

Adding a New Type Without Changing Existing Code

To support a brand-new payment method (e.g. adding PhonePe a year after launch), you simply create a new class that extends the same parent and overrides the relevant method with its own logic. The existing processor/router class, and all other existing payment classes, remain completely untouched. The client just needs to pass the new object type, and the same generic processing logic handles it correctly — this is the practical payoff of runtime polymorphism.

Why the Parameter Type Matters

The processor method's parameter must be declared as the parent (supertype), not a specific concrete type — this is what allows any child object to be passed in and have the correct overridden behavior invoked automatically. If the parameter were tied to one specific child type, the method would lose its generic, extensible nature entirely.

Class-Specific Logic Belongs Inside That Class, Not the Router

If some behavior is specific to only one particular subtype (e.g. a "welcome card" feature only relevant to card payments), that logic should live inside that specific class, not be crammed into the shared generic router/processor logic. The router stays generic and simple; specialization happens within each subclass.

Avoiding Duplicate Code Across Sibling Classes — "Has-A" (Aggregation)

If multiple sibling child classes (all extending the same parent) need to share some additional common method that isn't appropriate to place in the shared parent (since not every sibling needs it), and Java doesn't support multiple inheritance (a class can't extend two classes), one option is aggregation ("has-a" relationship): create an object of the class that has the needed method, and call it from within the class that needs it — reusing functionality without inheritance duplication.

💻 Code Snippets / Examples

Recap: notification example demonstrating runtime polymorphism:

class Notification {
    public void sendNotification() {
        System.out.println("Generic notification");
    }
}

class EmailNotification extends Notification {
    @Override
    public void sendNotification() {
        System.out.println("Sending email notification");
    }
}

class SMSNotification extends Notification {
    @Override
    public void sendNotification() {
        System.out.println("Sending SMS notification");
    }
}
Notification n1 = new EmailNotification();
n1.sendNotification(); // "Sending email notification" — decided at runtime

Notification n2 = new SMSNotification();
n2.sendNotification(); // "Sending SMS notification" — decided at runtime

Payment system — Payment as parent, multiple payment types as children:

class Payment {
    public void doPayment() {
        System.out.println("Generic payment");
    }
}

class UpiPayment extends Payment {
    @Override
    public void doPayment() {
        System.out.println("Doing UPI payment");
    }
}

class CardPayment extends Payment {
    @Override
    public void doPayment() {
        System.out.println("Doing card payment");
    }
}

class GPayPayment extends CardPayment {
    @Override
    public void doPayment() {
        System.out.println("Doing GPay payment");
    }
}

PaymentProcessor — generic router using polymorphism:

class PaymentProcessor {
    public void processPayment(Payment payment) {
        payment.doPayment(); // runtime polymorphism decides which version runs
    }
}
class Driver {
    public static void main(String[] args) {
        PaymentProcessor processor = new PaymentProcessor();

        Payment payment = new UpiPayment();
        processor.processPayment(payment); // "Doing UPI payment"

        payment = new CardPayment();
        processor.processPayment(payment); // "Doing card payment"
    }
}

Adding PhonePe a year later — NO changes to existing classes:

// NEW class added — nothing else touched
class PhonePePayment extends Payment {
    @Override
    public void doPayment() {
        System.out.println("Doing PhonePe payment");
    }
}
Payment payment = new PhonePePayment(); // client just sends the new type
processor.processPayment(payment); // "Doing PhonePe payment" — existing processor code unchanged

Aggregation ("has-a") to avoid duplicate code across siblings:

class Child2 {
    public void m2() {
        System.out.println("Child2's shared logic");
    }
}
class Child3 extends Parent {
    // Child3 needs Child2's m2() logic, but can't extend two classes
    private Child2 c2 = new Child2(); // aggregation — "has-a" relationship

    public void useSharedLogic() {
        c2.m2(); // reusing without duplicating the code
    }
}

📌 Important Points / Common Mistakes

  • Runtime polymorphism = method overriding, decided at runtime based on actual object type.
  • Compile-time polymorphism = method overloading, decided at compile time based on the method signature used.
  • The real-world value of polymorphism is enabling systems to be extended (new subclasses added) without modifying existing, tested, deployed code.
  • Generic processor/router methods should accept the parent type as their parameter to enable this extensibility.
  • Keep class-specific logic inside its own class rather than polluting shared/generic router logic.
  • Use aggregation ("has-a") to share functionality between sibling classes when multiple inheritance isn't possible.
