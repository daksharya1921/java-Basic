🧠 Key Concepts

Recap: Regular Class Structure

A normal class contains variables (state) and methods (fully implemented functionality/behavior) — both the what a method does and how it does it are defined together in one place.

What is an Abstract Class?

An abstract class separates a method into two conceptual parts: what needs to be done, and how it should be done. The abstract class declares what — it doesn't specify how. It's up to the child classes to decide how to actually implement that functionality.

Declaring an Abstract Class and Abstract Method 
abstract class PaymentSystem {
    public abstract void doTransaction(); // no body — ends with semicolon
}

The abstract keyword marks the class as abstract. An abstract method has no body at all — just a signature ending in a semicolon. It only declares what must be done; how is left entirely to child classes.

Abstract Classes Force Child Classes to Implement

Any class that extends an abstract class must provide an implementation (a body) for all its inherited abstract methods — otherwise the code won't compile. This is how an abstract class enforces a contract: "if you want to be my child, you must implement these specific functionalities."

Abstract Classes Can Mix Abstract and Concrete Methods

An abstract class isn't required to make every method abstract. It can contain a mix: some methods fully implemented (concrete methods, with a body) that are common/shared functionality inherited as-is, and other methods left abstract because their implementation genuinely varies per child class.

Why Use Abstract Classes When Regular Classes Work Fine?

When designing a system (e.g. as an architect), you often know what functionality must exist across all variations (e.g. every payment type must support "do transaction" and "suspend transaction"), but you don't know or care how each specific variation (UPI, card, wallet) will implement it internally. Abstract classes let you set up this structural contract cleanly — the class declaring the requirement doesn't need to know implementation details of classes that don't exist yet.

Cannot Instantiate an Abstract Class

You cannot create an object of an abstract class directly using new — because it's only partially implemented (it has methods with no bodies). Java prevents instantiating something that isn't fully functional.

Constructors Still Exist in Abstract Classes

Even though you can't instantiate an abstract class, it still has a constructor (default, if none is written) — this is necessary to maintain constructor chaining. When a child class object is created, its constructor implicitly calls super(), which runs the abstract class's constructor — useful for initializing any state the abstract class defines.

Abstract vs Final — Key Interview Question
abstractfinal
Class MUST be extended to be usefulClass CANNOT be extended at all

Because of this direct contradiction, a class can never be declared both abstract and final at the same time — abstract requires that it become a parent, while final forbids it from ever becoming a parent.

Designing with Abstract Classes — The Architect's Perspective

Typically, the person designing the overall system structure (architect) decides which functionalities are mandatory for all variations (made abstract) and which are common/shared across all variations (made concrete, provided directly). Individual developers then build the specific child classes, implementing only the abstract methods relevant to their specific variation.

💻 Code Snippets / Examples

Abstract class with mixed abstract and concrete methods:

abstract class PaymentSystem {

    public abstract void doTransaction();     // abstract — no implementation
    public abstract void suspendTransaction(); // abstract — no implementation

    public void sendSMS() { // concrete — shared/common functionality
        System.out.println("Sending SMS notification");
    }

}

Child class must implement all abstract methods:

class UpiPayment extends PaymentSystem {

    @Override
    public void doTransaction() {
        System.out.println("Doing UPI transaction");
    }

    @Override
    public void suspendTransaction() {
        System.out.println("Suspending UPI transaction");
    }

    // sendSMS() inherited as-is — no need to reimplement

}

Cannot instantiate an abstract class:

// PaymentSystem ps = new PaymentSystem(); // COMPILE ERROR — cannot instantiate abstract class

PaymentSystem ps = new UpiPayment(); // valid — upcasting a concrete child object
ps.doTransaction();
ps.sendSMS();

Constructor chaining still works with abstract classes:

abstract class PaymentSystem {
    public PaymentSystem() {
        System.out.println("PaymentSystem constructor called");
    }
    public abstract void doTransaction();
}
class UpiPayment extends PaymentSystem {
    public UpiPayment() {
        // implicit super() call happens first
        System.out.println("UpiPayment constructor called");
    }

    @Override
    public void doTransaction() {
        System.out.println("UPI transaction done");
    }
}
UpiPayment upi = new UpiPayment();
// Output:
// PaymentSystem constructor called
// UpiPayment constructor called

Practice exercise — E-commerce Order Management using abstract class:

abstract class OrderManagement {

    public abstract void placeOrder();
    public abstract void editOrder();
    public abstract void confirmOrder();

    public void getOrderStatus() { // concrete — common to all order types
        System.out.println("Fetching order status");
    }

}
class RetailUser extends OrderManagement {
    @Override
    public void placeOrder() {
        System.out.println("Retail user placing order");
    }

    @Override
    public void editOrder() {
        System.out.println("Retail user editing order");
    }

    @Override
    public void confirmOrder() {
        System.out.println("Retail user confirming order");
    }
}
class PremiumUser extends OrderManagement {
    @Override
    public void placeOrder() {
        System.out.println("Premium user placing order with priority");
    }

    @Override
    public void editOrder() {
        System.out.println("Premium user editing order");
    }

    @Override
    public void confirmOrder() {
        System.out.println("Premium user confirming order with instant confirmation");
    }
}

❓ Doubts / Questions Raised in Class

QuestionAnswer
Why not just put full implementation in one class instead of using abstract?In real systems, the architect designing the structure often doesn't know (or care) how each specific variation will implement certain functionality — abstract classes let you separate "what must exist" from "how it's implemented," which different developers/teams handle for each child class.
Why can't you create an object of an abstract class?Because it's only partially implemented — some methods have no body, so an instance of it wouldn't be fully functional.
If you can't instantiate an abstract class, why does it need a constructor?To support constructor chaining — when a child class object is created, the abstract class's constructor is still called via implicit super(), useful for initializing shared state.
Can a class be both abstract and final?No — abstract requires the class to be extended (become a parent), while final forbids exactly that; the two are mutually exclusive.
Where would abstract classes be used in a real project?Whenever a system needs to guarantee that certain functionality will be implemented by all variations/subtypes, while allowing each to implement it differently (e.g. different payment methods all needing to support "do transaction," each with its own internal logic).

📌 Important Points / Common Mistakes

  • Abstract methods have no body — just a signature ending in a semicolon.
  • Any concrete (non-abstract) class extending an abstract class MUST implement all inherited abstract methods.
  • Abstract classes can freely mix abstract methods (must be implemented by children) and concrete methods (shared, inherited as-is).
  • You cannot instantiate an abstract class directly, but you can still upcast a concrete child object to the abstract type.
  • A class can never be both abstract and final — these modifiers directly contradict each other.
  • Common/shared functionality that doesn't vary per child should be a concrete method in the abstract class, not duplicated across every child.
