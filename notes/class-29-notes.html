Class 29 — Method Overriding Rules & Covariant Return Types

🎯 Topics Covered

  • Recap: upcasting/downcasting, method overriding basics
  • Private and final methods cannot be overridden
  • Full rules for valid method overriding
  • Covariant return types (introduced in Java 5) — detailed explanation
  • Overriding requires a parent-child (inheritance) relationship
  • Only one public class allowed per Java file
  • Static method "hiding" (brief mention, not full overriding)

🧠 Key Concepts

Private Methods Cannot Be Overridden

Overriding requires a method to be visible outside its own class (since overriding happens between parent and child classes). Since private methods are only visible within their own declaring class, they cannot participate in overriding at all — a child class simply can't "see" them to override.

final Methods Cannot Be Overridden

Marking a method final in the parent class prevents any child class from overriding it — while the method remains fully usable (callable) in child classes exactly as inherited, its implementation cannot be changed. This gives fine-grained control: you can allow some methods in a class to be overridden while locking others.

final vs private — Key Difference

Both restrict overriding, but differently: a private method isn't even visible/usable outside its class. A final method IS visible and usable (inherited and callable) in child classes — it just can't be given new behavior via overriding.

Rules for Valid Method Overriding
1️⃣ Method name — must match exactly (Java is case-sensitive)
2️⃣ Parameters — same order, same types, same number
3️⃣ Return type — same, or covariant (see below)
4️⃣ Access modifier — same or wider (never narrower)
5️⃣ Exceptions — specific rules (covered later, once exception handling is taught)
Rule 1: Method Name Must Match Exactly

Even a single differing letter case (e.g. calculatePrice vs CalculatePrice) makes it a completely different, unrelated method in Java's eyes — not an override. Java is strictly case-sensitive.

Rule 2: Parameters Must Match

The overriding method must have the same number, same types, and same order of parameters as the parent method. Parameter variable names don't matter (Java doesn't check those) — only their types, count, and order.

Rule 3: Access Modifier — Same or Wider Only

The overriding method's access modifier must be at least as visible as the parent's — you can keep it the same or make it wider, but you can never narrow it. Example: default → default (allowed), default → protected/public (allowed, widening), but protected → default (NOT allowed, narrowing).

Why this rule exists: If narrowing were allowed, a child class could override a public method and make it private — effectively blocking any further child classes from being able to override or even use it, breaking the inheritance chain's consistency.

Why Override Instead of Writing a New Method?

Even if the internal logic is completely different, overriding maintains a consistent contract/interface across all related classes. Example: if every subclass of Customer needs to calculate a price, they should all expose the same method name (calculatePrice()) — even with different internal logic — rather than each having a differently-named method, which would break consistency and make the codebase harder to use predictably (similar to why every class inherits a consistently-named equals() method from Object, even though each class implements its comparison logic differently).

Rule 4 (Covariant Return Types) — What "Covariant" Means

Introduced in Java 5. Normally, the overriding method's return type must be exactly the same as the parent's. Covariant return type is a special exception: the overriding method in the child class can return either the same type OR a child (subtype) of the parent method's declared return type.

Covariant Return Type — Worked Example

If a parent class method's return type is Post, an overriding method in a child class can return either Post itself, or any class that IS-A Post (e.g. SpeedPost, if SpeedPost extends Post). This only works because a genuine parent-child relationship exists between the two return types — if that relationship doesn't exist, covariant returns don't apply, and the return type must be exactly the same.

One Public Class Per Java File

A single .java file can contain multiple classes, but only one of them may be declared public — and if there is a public class, the file name must exactly match that class's name. Multiple public classes in one file cause a compilation error.

Static Methods — "Hiding," Not True Overriding (Brief Mention)

If a subclass declares a static method with the same signature as a static method in the parent, this is technically called method hiding, not overriding — because static members belong to the class, and true overriding is resolved at runtime based on the actual object, which doesn't apply the same way to static context. This was flagged as a topic to be covered in more depth later.

💻 Code Snippets / Examples

Private methods cannot be overridden:

class Customer {
    private void internalDetails() {
        System.out.println("internal only");
    }

    public void calculatePrice() {
        System.out.println("Customer price logic");
    }
}
class GoldCustomer extends Customer {
    // internalDetails() is NOT visible here — cannot be overridden

    @Override
    public void calculatePrice() { // valid override — public, visible
        System.out.println("Gold customer discounted price logic");
    }
}

final method — cannot be overridden, but still usable:

class Customer {
    public final void loyaltyPoints() {
        System.out.println("Standard loyalty points logic");
    }
}
class GoldCustomer extends Customer {
    // @Override
    // public void loyaltyPoints() { }  // ERROR: cannot override final method
}
GoldCustomer gc = new GoldCustomer();
gc.loyaltyPoints(); // works fine — inherited as-is, just can't be overridden

Access modifier — allowed widening, disallowed narrowing:

class Customer {
    protected void printDetails() {
        System.out.println("Customer details");
    }
}
class GoldCustomer extends Customer {

    @Override
    public void printDetails() { // ALLOWED — protected widened to public
        System.out.println("Gold customer details");
    }

}
// NOT ALLOWED:
// @Override
// private void printDetails() { }  // ERROR: cannot reduce visibility from protected to private

Covariant return type — full worked example:

class Post { }

class SpeedPost extends Post { } // SpeedPost IS-A Post
class Delivery {
    public Post doDelivery() {
        return new Post();
    }
}
class OneDayDelivery extends Delivery {

    @Override
    public SpeedPost doDelivery() { // covariant — SpeedPost is a child of Post
        return new SpeedPost();
    }

}

Without a valid parent-child relationship, covariant return doesn't apply:

class Post { }
class UnrelatedClass { } // no inheritance relationship with Post
class Delivery {
    public Post doDelivery() {
        return new Post();
    }
}
class OneDayDelivery extends Delivery {
    // @Override
    // public UnrelatedClass doDelivery() { }
    // ERROR: UnrelatedClass is not a subtype of Post — covariant rule doesn't apply
}

Object as return type — anything can be returned covariantly:

class Delivery {
    public Object doDelivery() {
        return new Object();
    }
}
class OneDayDelivery extends Delivery {
    @Override
    public Post doDelivery() { // valid — Post is a child of Object (everything is)
        return new Post();
    }
}

❓ Doubts / Questions Raised in Class

QuestionAnswer
Why override a method instead of just writing a differently-named new method with the desired logic?To maintain a consistent method contract/interface across related classes — the same purpose should be accessible via the same method name, regardless of differing internal implementations.
Why can't a child class narrow the access modifier when overriding?If allowed, a child could override a public method as private, effectively blocking further subclasses from accessing or re-overriding it, breaking the consistency the parent originally established.
What exactly is covariant return type?The overriding method can return the same type as the parent's declared return type, OR any valid subtype (child class) of it — this flexibility was introduced in Java 5.
Does covariant return type work without any parent-child relationship between the two return types?No — a genuine inheritance relationship must exist between the original return type and the new return type for covariant returns to be valid.

📌 Important Points / Common Mistakes

  • Private methods can never be overridden — they're not visible to child classes at all.
  • final methods prevent overriding but remain fully usable/inherited as-is.
  • Method name, parameter types/order/count must match exactly for valid overriding.
  • Access modifiers can only stay the same or widen when overriding — never narrow.
  • Covariant return types (Java 5+) allow returning a subtype of the parent method's declared return type — but only when a real inheritance relationship exists between the types.
  • Only one public class is allowed per .java file, and it must match the file name exactly.
  • Return type is not limited to primitives/void — it can be any valid object type, including custom classes.
