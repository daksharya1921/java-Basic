🧠 Key Concepts

Why Code Must Be Flexible

A product/application (e.g. a ride-booking app) can remain in the market for many years (instructor used 15 years as an example). Since it will be used continuously, code should be written so that future changes don't require modifying the Java code itself.

Code vs Configuration

A good developer separates Code from Configuration. Configuration refers to values that may change in the future — such as currency type, interest rate, or file paths. These should not be hardcoded inside the Java code; they should be read from a configuration file, command line argument, or database instead.

Example: Currency

If a payment module hardcodes the currency as "INR", expanding the application to another country (e.g. supporting USD or EUR) would require a Java code change, recompilation, and a full testing cycle. Reading the currency from a configuration source instead allows the value to be updated without touching the Java code.

Properties File (Configuration Example)

A configuration file (e.g. rapido.properties) stores values as key-value pairs alongside the Java code. To change a value, you update the file and restart the application — no code change, no recompilation needed.

Why Avoiding Hardcoding Saves Time

Changing hardcoded Java code requires: code change → compile → unit testing → QA testing → system integration testing → pre-prod → production. This cycle can take 5–7 days or more. Reading from configuration instead can reduce the same change to a two-minute task (update value + restart application).

Example: Inconsistent File Names

When multiple people submit data files (e.g. marketing/sales data from different cities), each person may name their file differently. A program should not hardcode an expected file name — it should accept the file name flexibly (e.g. via command line argument) so it can process whatever file name is actually given.

Java Identifiers

An identifier is the name given to a class, variable, or method — it is how you uniquely refer to that class/variable/method in code. Example: in class DataProcessor, DataProcessor is an identifier; in a variable declaration, the variable name is an identifier.

Rules for Identifiers
  • Allowed characters: uppercase letters (A–Z), lowercase letters (a–z), digits (0–9), underscore (_), and dollar sign ($)
  • An identifier cannot start with a digit (e.g. 1DataProcessor is invalid, but DataProcessor1 is valid)
  • It can start with an underscore or dollar sign (e.g. _DataProcessor or $DataProcessor are valid)
  • Other special characters (e.g. &, %) are not allowed in identifiers
  • If an identifier is missing where expected, the compiler shows the error: identifier expected
Reserved Keywords

Java has 55+ reserved keywords — words with a predefined meaning used by the compiler and JVM (e.g. class, public, static, void, int). These cannot be used as identifiers (cannot be used as a class, method, or variable name), since the compiler relies on them to understand code structure.

Reserved Keywords Are Always Lowercase

All Java reserved keywords are written in lowercase. This means a word like Public (capital P) is not a reserved keyword and can legally be used as an identifier — because Java is case-sensitive (public and Public are treated as different words).

How the Compiler Uses Keywords to Parse Structure

The compiler recognizes the overall structure of a class using reserved keywords. For example, after the class keyword, it expects a class name (identifier), followed by either extends, implements, or an opening curly brace { — not another arbitrary name directly. This is how the compiler validates and understands code structure.

💻 Code Snippets / Examples

Bad practice — hardcoded configuration value:

String currency = "INR"; // hardcoded — bad practice

Better practice — read from configuration:

String currency = readFromConfig("currency"); // read from configuration file/database

Example properties file (rapido.properties):

currency=INR

Bad practice — hardcoded file name:

class DataProcessor {

    public static void main(String[] args) {
        String fileName = "data_08_June.xls"; // hardcoded — not flexible
        System.out.println("Processing " + fileName);
    }

}

Better practice — file name from command line argument:

class DataProcessor {

    public static void main(String[] args) {
        String fileName = args[0]; // flexible — supplied at runtime
        System.out.println("Processing " + fileName);
    }

}
java DataProcessor June_1st_week_data.xls

Identifier example — this fails to compile:

= 20; // missing identifier (variable name) — compiler error: "identifier expected"

Correct:

int amount = 20; // "amount" is the identifier

Reserved keyword used incorrectly as a method name — fails to compile:

public public() {  } // "public" is a reserved keyword, cannot be used as identifier

Valid — because Java is case-sensitive:

public Public() {  } // "Public" (capital P) is NOT a reserved keyword, so this compiles

📌 Important Points / Common Mistakes

  • Never hardcode values that are likely to change — use configuration, command line arguments, or database values instead.
  • Identifiers can contain letters, digits, underscore, and dollar sign — but cannot start with a digit.
  • Reserved keywords are always lowercase and cannot be used as identifiers; capitalized versions of keywords are valid identifiers since Java is case-sensitive.
  • The compiler uses reserved keywords to understand code structure (e.g. what follows the class keyword).
  • A missing identifier where one is expected produces the compiler error: identifier expected.
