🧠 Key Concepts

JDK — Java Development Kit

JDK provides the compiler (javac). Without JDK, you cannot compile a Java program. When you install Java, you get JDK, and JDK internally includes JRE as well.

JRE — Java Runtime Environment

JRE provides the runtime environment needed to run (execute) a Java program. It supplies supporting/inbuilt classes such as String, System, and Scanner, which are part of the JRE's library — bundled inside a file called rt.jar.

What is a JAR file?

JAR stands for Java Archive — similar to a ZIP file, it bundles many compiled .class files together into one file. rt.jar (runtime jar) contains most of the general-purpose classes needed to run a Java program, organized into packages (e.g. java.lang contains common classes like String and System).

JDK vs JRE — Where Each is Installed

On a development machine, JDK is required (since code needs to be compiled). On a QA/testing or production machine, only JRE is required — since that team only runs the already-compiled code, they don't need the compiler.

  • JDK → required for development (compiling code)
  • JRE → required for testing/running (executing code)
JVM — Java Virtual Machine

Unlike JDK and JRE (which are physical/installable), JVM is a virtual machine — a process that starts only when you run a program (via the java command). It occupies memory (MetaSpace, Heap, Stack), loads the required class, executes the program, and once the main method finishes, JVM shuts down and releases all occupied memory.

Bytecode, Interpreter & JIT Compiler

When a Java program runs, the class is loaded as bytecode (not directly understood by the computer/hardware — only native machine code is). Inside JVM, there is an interpreter and a Just-In-Time (JIT) compiler that work together to convert bytecode into native machine code.

  • The interpreter converts and executes code line by line.
  • For repeated code (e.g. a loop running thousands of times), the JIT compiler optimizes execution instead of interpreting the same line repeatedly — this improves performance for frequently executed code.
Why Avoid Hardcoding

Hardcoding means directly writing fixed values into code (e.g. int interestRate = 5;). This is a bad practice because if the value needs to change later (e.g. interest rate becomes 6), the code must be modified, recompiled, tested (unit testing → QA → system integration testing → production) — a process that can take 15 days to a month, and risks introducing new bugs elsewhere in the system.

Command Line Arguments — Avoiding Hardcoding

Instead of hardcoding values, input can be supplied from the command line while running the program, using String[] args. This makes the program flexible — values can change without modifying or recompiling the code.

Command Line Arguments are Always Strings

All command line arguments are stored as String, even if you pass a number. So args[0] holding "1" is a string "1", not the integer 1. Using the + operator directly on string numbers performs concatenation, not addition (e.g. "1" + "1" gives "11", not 2).

Converting String to Integer

To perform mathematical operations on command line input, convert the string to an integer using the built-in Integer.parseInt() method, which takes a String and returns an int.

Handling Spaces in Arguments

Space is the separator between command line arguments. If a value itself contains a space (e.g. a full name), wrap it in double quotes so it is treated as a single argument instead of being split into two.

Naming Convention Recap

Variable naming convention is the same as method naming convention — camelCase (first word lowercase, subsequent words capitalized). This applies consistently across class, method, and variable naming, each following their respective convention.

💻 Code Snippets / Examples

class Employee {

    public static void main(String[] args) {
        System.out.println("Employee details");
    }

}

Extracting rt.jar contents (demonstrated via jar command):

jar xf rt.jar

Hardcoded version (bad practice):

class AddNumbers {

    public static void main(String[] args) {
        int sum = 5 + 20; // hardcoded values
        System.out.println(sum);
    }

}

Using command line arguments instead:

class AddNumbers {

    public static void main(String[] args) {
        String firstNumberStr = args[0];
        String secondNumberStr = args[1];

        int firstNumber = Integer.parseInt(firstNumberStr);
        int secondNumber = Integer.parseInt(secondNumberStr);

        int sum = firstNumber + secondNumber;
        System.out.println(sum);
    }

}
java AddNumbers 20 20

Handling a name with spaces:

java Employee "Pooja Academy"

📌 Important Points / Common Mistakes

  • JDK = development (has compiler); JRE = runtime (needed to execute code); both are physically installable, unlike JVM.
  • JVM is a virtual machine/process — it starts when the program runs and shuts down once execution completes.
  • Never hardcode values that are likely to change — use command line arguments instead.
  • All command line arguments are received as String, even numbers — convert explicitly using Integer.parseInt() before doing math.
  • Use double quotes to pass a value containing spaces as a single command line argument.
  • Running a program that expects command line arguments without supplying them causes an index-out-of-bounds error.
  • JIT compiler improves performance specifically for repeated/hot code paths, working alongside the interpreter.
